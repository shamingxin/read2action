## 1. 核心方法论总览（一句话）

**先用 mock + 分阶段交付跑通主链路，再以走查表 + 严重程度分级收口；文档与浏览器事实同源，旧规格文档降级为「对照参考」而非唯一真理。**

---

## 2. 分阶段交付与「不要一次做完」

### 2.1 为何分阶段

- **降低上下文丢失风险**：每阶段文件与验收边界清晰，便于 Cursor 新对话接力。  
- **控制返工成本**：先结构后视觉，先主链路后占位，避免一上来接后端/AI 导致全栈纠缠。  
- **对齐 Figma 成本**：高保真 6 页分步落地，比「一次生成全站」更易对齐组件与 token。

### 2.2 实践要点（来自材料清单与项目实际）

- 每一阶段结束时要求 Agent **列出改动文件、测试方式、遗留问题**。  
- **禁止**单次对话「把整个项目做完」的指令风格。  
- **1.0 完成后**：进入 **1.1 准备期** 时反过来 **冻结代码**，先对齐接口与立项 — 这是 **另一种分阶段**（方案先行）。

## 3. Mock-first 与边界意识

### 3.1 Mock-first 路径

```text
mock data → local state → simulated parsing → 可部署演示 →（1.1）真实 API →（2.0）持久化与账号
```

### 3.2 边界写清「不接什么」

明确 **不做登录、不做真实数据库、不做真实 AI（1.0）** 等，可减少 Agent 自作主张引入 Supabase/OAuth 等。**1.0 上线后**仍需反复强调：**线上 mock 非 bug**，避免误报「假数据问题」。

## 4. 第九阶段 9.3：主流程走查方法论

### 4.1 阶段原则（原文保留）

- **不接真实 API**、**不改 mock 数据语义**、**不大改布局**、**不做 2.0 视觉优化**。  
- 验收以 **最新接力文档 + 浏览器实际效果** 为准，**不以**旧版《Read2Action_交互跳转逻辑文档》v0.7 为唯一标准。  
- **9.3 不是新功能开发阶段**：定位是 **走查、登记、极小修复、复验**。

## 5. 视觉与工程：Token 化（9.2 复盘）

### 5.1 做法

将 **页面壳、卡片圆角/阴影、顶栏、Tab 下划线** 等收敛到 **`src/lib/r2a-ui-classes.ts`**（历史文件名，以仓库为准），避免每页复制 Tailwind 长串。

### 5.2 收益

- **跨页一致性**：`/result`、笔记详情、`/parsing`、项目列表、首页纳入同一 **1020** 体系。  
- **后续细修成本下降**：改 token 即可批量影响多页。

### 5.3 代价与注意

- **非像素级一致**仍可能存在（字体、内部 max-w 等），需 **P2/P3** 队列管理，避免在 9.3 无限放大范围。

## 7. 已知「稿差 / 占位」清单（方法论：如何不把它当事故）

以下项来自接力文档 **§7**，属 **预期内技术债** 而非「上线失败」：

- 字体与稿偏差；间距/圆角/阴影非处处像素级。  
- 日期格式不统一（列表 vs 详情）。  
- 内部 `max-w` 与 1020 壳并存。  
- **方法论沉淀** UI 文案仍为占位。  
- **`/result` 原文对照 Tab** 仍为占位。  
- **取消解析确认弹窗** 可能与早期交互稿不一致。

**处理方法**：在 `03_下一步任务清单.md` 建 **体验专项** 或排入 **1.1 之后**，并标注 **不阻塞 1.0 / 不默认进 1.1 首包**。

## 9. 版本路线决策的可复用模板（1.0 / 1.1 / 2.0）

当「当前不做」清单过长时，容易与 **下一阶段** 混淆。本项目采用：

- **1.0**：冻结范围 + **上线事实**（mock 可接受）。  
- **1.1**：**单一清晰增量**（例如仅真实 AI + `/api/analyze`），仍排除登录/DB。  
- **2.0**：商业化与数据层能力打包为 **远期**，避免 1.1 scope creep。
