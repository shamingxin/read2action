## 2. 分阶段交付与「不要一次做完」

### 2.1 为何分阶段

- **降低上下文丢失风险**：每阶段文件与验收边界清晰，便于 Cursor 新对话接力。  
- **控制返工成本**：先结构后视觉，先主链路后占位，避免一上来接后端/AI 导致全栈纠缠。  
- **对齐 Figma 成本**：高保真 6 页分步落地，比「一次生成全站」更易对齐组件与 token。

### 2.2 实践要点（来自材料清单与项目实际）

- 每一阶段结束时要求 Agent **列出改动文件、测试方式、遗留问题**。  
- **禁止**单次对话「把整个项目做完」的指令风格。  
- **1.0 完成后**：进入 **1.1 准备期** 时反过来 **冻结代码**，先对齐接口与立项 — 这是 **另一种分阶段**（方案先行）。

### 2.3 1.1 接力经验：服务端 API 先独立验证，再接前端页面

- **做法**：在 **`08`** 契约对齐后，**先**实现 **`POST /api/analyze`** 等 **Route Handler**，用 **`curl` / Postman** 在本地或预览环境 **独立打通**模型与 JSON 映射；**再**进入 **`/parsing` / `/result`** 接线。  
- **收益**：密钥与上游错误在 **小范围** 内暴露、易排查；**避免**「页面 + 动画 + 路由 + AI」同时失败时难以归因；符合 **密钥仅服务端** 的安全习惯。  
- **配套**：**`.env.local` 不入库**；**`.env.example` + README** 写清变量名；**每小步列文件清单并经确认后再写代码**，防止 scope creep。

### 2.4 1.1 真实 AI 接入的工程纪律：防重、超时、取消、费用与 Network 验证

- **请求防重**：前端 **`/parsing`** 等页面 **不得**把 **`POST /api/analyze`** 绑在会因 **`setState` / `router` 引用变化** 而反复触发的 **`useEffect` 依赖链**上；应 **单次挂载启动 + 世代门闩（如 `commitRef`）+ `inFlightRef`**，并 **兼容 React StrictMode**（dev 下 effect 双跑 **不得**变成双计费）。**禁止**自动无限重试；**仅用户显式「重试」** 才发新请求。  
- **超时**：客户端 **`fetch` 须有上限**（本项目 **`postAnalyze` 默认 30s**），超时映射为可读错误，**避免**长挂连接与不可控账单。  
- **取消**：**`AbortController.abort()`** 与 **清定时器** 必须同时到位；取消后 **不得**再跳转或写成功态；**pending 正文 session key** 须在 **成功 / 失败 / 取消** 等终态清理，避免再次进入 **`/parsing`** 自动重打 API。  
- **费用控制意识**：真实供应商 **按次 / 按 token** 计费；P0 级事故多为 **循环请求** — 接 AI 第一天就要把 **「单次用户动作 → 最多 N 次上游调用」** 写进验收与自测。  
- **Network 先于盲测真 API**：先用 **`npm run test`（mock `fetch`）** 或本地代理确认 **计数**，再用浏览器 **DevTools → Network** 确认 **单次主路径仅 1 条 `POST /api/analyze`（未点重试）**，**然后** 才做短时、可监督的真 API 抽检；**禁止**未验证即对生产供应商做无监督、长时间或脚本化压测。

## 3. Mock-first 与边界意识

### 3.1 Mock-first 路径

```text
mock data → local state → simulated parsing → 可部署演示 →（1.1）真实 API →（2.0）持久化与账号
```

### 3.2 边界写清「不接什么」

明确 **不做登录、不做真实数据库、不做真实 AI（1.0）** 等，可减少 Agent 自作主张引入 Supabase/OAuth 等。**1.0 上线后**仍需反复强调：**线上 mock 非 bug**，避免误报「假数据问题」。

## 5. 视觉与工程：Token 化（9.2 复盘）

### 5.1 做法

将 **页面壳、卡片圆角/阴影、顶栏、Tab 下划线** 等收敛到 **`src/lib/r2a-ui-classes.ts`**（历史文件名，以仓库为准），避免每页复制 Tailwind 长串。

### 5.2 收益

- **跨页一致性**：`/result`、笔记详情、`/parsing`、项目列表、首页纳入同一 **1020** 体系。  
- **后续细修成本下降**：改 token 即可批量影响多页。

### 5.3 代价与注意

- **非像素级一致**仍可能存在（字体、内部 max-w 等），需 **P2/P3** 队列管理，避免在 9.3 无限放大范围。

### 11.4「AI 是否提效」— 阶段性判断（诚实口径）

- **提效明显的阶段**：把 **规格写成文档**、把 **大任务拆成小步**、把 **重复性 UI / 样板代码** 交给 Cursor 时，**单位时间产出** 通常高于纯手写。  
- **提效有限或需加成本的阶段**：**长对话** 导致 **上下文漂移**、**需求在聊天里改口** 但未同步进 docs 时，Agent **容易做错或做多**；纠偏与返工会吃掉收益。  
- **结论**：1.0 的提效 **高度依赖「文档真值 + 小步验收」**；**不是**「只要 AI 写就会快」。

### 11.5 1.0 暴露的问题（复盘承认，用于 1.1 与下一项目）

1. **文档分散**：早期多份 `Read2Action_*` 并行，**接力成本高**；已通过 **01～07 固定中文文档体系** 收敛（见 `06`）。  
2. **长对话上下文成本**：Cursor 单会话过长时，**隐含假设丢失**、重复解释多；对策是 **新开对话 + @ docs**，而非堆在同一线程。  
3. **Cursor 容易自作主张**：在边界未写死时，倾向 **加依赖、加功能**；需 **明确「不做」+ 小 PR / 小提交** 约束。  
4. **人工验收不可替代**：**构建通过 ≠ 产品对**；主流程、Toast、侧栏、线上与本地一致性等，仍以 **人眼 + 轻量走查表** 为准（9.3 方法论即来源之一）。  
5. **1.0 仍是 mock 演示版**：上线价值在 **可演示、可部署、边界诚实**；**不应**包装成「已具备真实 AI / 真实数据」的产品，以免作品集与面试叙事 **失真**。

### 11.6 可复用方法论（从 1.0 复盘沉淀）

| 方法 | 说明 |
|------|------|
| **先闭环再高级** | 先 **mock 主链路 + 部署可点**，再考虑真实 AI、持久化、账号等；避免「半成品堆功能」。 |
| **docs 管理上下文** | **共识进 `01`～`07`**，对话只作 **临时推导**；减少「口头需求」与仓库真值偏离。 |
| **小任务驱动 Cursor** | 单次指令 **可验证、可回滚**；拒绝「一次做完整个项目」。 |
| **重要节点 Git 提交** | 阶段验收、文档体系变更、范围冻结等 **单独 commit**，便于 diff 与责任边界。 |
| **阶段性防自嗨检查** | 周期性自问：**是否超出 PRD？是否把 mock 当 bug？是否把 Agent 输出当真理？** 与 **人工验收** 绑定。 |
