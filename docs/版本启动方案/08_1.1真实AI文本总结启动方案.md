## 2. 1.1 只做

在 **既有六页主流程** 下，**仅**实现以下能力（与 `02` §8、`03`、`04` **2026-05-16** 一致）：

1. **用户输入**：以 **纯文本** 作为 **1.1 唯一可靠输入**（长度上限在实现与契约中约定，见下文 API）。  
2. **调用 AI**：经 **后端 Route Handler**（推荐路径 **`/api/analyze`**，或等价路径但须在仓库内统一命名）调用 **单一供应商优先** 的大模型，生成 **结构化总结**。  
3. **结果展示**：将模型输出 **映射** 到现有 UI 所需结构（与当前 **`ParseResultPreview`** 及保存链路字段对齐），在 **`/result`** **真实展示** AI 生成内容（替换当前 mock 的「核心解析结果」体验）。  
4. **`/parsing` 四步**：**保留** 四步加载的 **产品体验**；四步与真实请求耗时的衔接方式（假进度 / 阶段事件 / 最短展示时间）在实现时确定，但 **不得** 以此为理由扩大 1.1 产品范围。  
5. **链接形态**：**不抓取**；仅允许 **轻量提示** 管理预期（见 **§9**）。  
6. **保存到项目**：在 **不做数据库** 前提下，延续 **1.0 既有前端状态 / localStorage 等机制** 将「当前一次解析结果」写入「沙」等（具体是否扩展字段以立项时对照代码定稿）；**禁止**借机引入服务端落库、账号体系。

## 4. 用户流程

### 4.1 主流程（与 1.0 一致，结果来源由 mock 换为真实 AI）

```text
首页（/）输入纯文本
  → 点击「开始解析」
  → /parsing：四步进度展示（与真实请求并行或串行策略由实现定稿）
  → /result：展示真实 AI 结构化总结（一句话总结 / 核心观点 / 行动清单 / 知识卡片等字段）
  → 保存到项目（仍走前端状态机制，不落库）
  → /projects/sha 列表可见
  → /projects/sha/notes/[id] 详情（内容总结 / 原文对照 Tab）
```

### 4.2 分支：输入被判定为「主要为链接」

- **不发起**抓取；**不调用**「读 URL」类能力。  
- **行为**：阻止或引导用户改为粘贴正文（见 **§9**）；若产品允许「仍当普通文本交给模型」，须在方案中明确 **风险与免责声明**，且 **默认推荐** 仍以「提示 + 要求正文」为主，与 `02`/`03` 口径一致。

### 4.3 分支：解析失败

- 用户仍停留在 **`/parsing`** 或 **`/result`** 的 **可理解错误态**（见 **§10**），可回首页重试；**不提供**「换账号 / 同步云端历史」等 2.0 能力。

## 5. `/api/analyze` API 路线

### 5.1 技术选型（建议）

- **运行时**：Next.js **App Router** 的 **Route Handler**（如 `src/app/api/analyze/route.ts`），**仅服务端** 读取模型 API Key（环境变量 / Vercel Secrets）。  
- **方法**：**`POST`**；**不支持**在浏览器直连第三方大模型（避免密钥泄露）。  
- **内容类型**：`application/json`；UTF-8。  
- **幂等**：单次请求独立；**不做**用户级配额服务端的复杂实现（若需限流，优先 **Vercel / 供应商** 侧能力与简单中间层，且不引入账号系统）。

### 5.2 与页面协作（逻辑顺序）

1. 前端在 **`/parsing`** 或 **进入 `/parsing` 前** 将 **纯文本**（及可选 **客户端生成 requestId**）提交给 **`POST /api/analyze`**。  
2. 服务端校验 → 调用模型 → 返回 **结构化 JSON**（见 **§6、§7**）。  
3. 前端将结果写入 **内存状态 / URL 状态 / localStorage**（与 1.0 现有数据流对齐，具体以代码审计为准），再进入 **`/result`** 渲染。  
4. **`/parsing` 四步**：可与请求 **并行**（先发起请求，四步仅展示节奏）或 **串行**（四步完成后再请求）—— **须在开发前在本文档或 PR 描述中选定一种并写入验收**。

### 5.3 链接检测（产品层）

- **建议位置**：**首页提交前** 与 **服务端二次校验**（防绕过）；服务端检测到「主要为 URL」时返回 **可区分错误码**（见 **§6**），前端映射为 **§9** 文案。

## 6. 请求 / 返回 JSON 契约

> 以下为 **立项草案**；字段可在开发前微调，但 **不得** 通过加字段引入抓取、账号、落库等超出 **§3** 的能力。

### 6.1 请求体 `AnalyzeRequest`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `text` | `string` | 是 | 用户粘贴的 **纯文本**；服务端校验 **非空**、**最大长度**（建议初版 **8k～32k 字符** 间取一值，按模型成本复评） |
| `locale` | `string` | 否 | 输出语言偏好，如 `zh-CN`；缺省由 Prompt 约束 |
| `requestId` | `string` | 否 | 客户端生成的 UUID，便于日志关联；**不落库** |

**服务端拒绝示例**：

- `text` 为空 → `400`，`code: "EMPTY_INPUT"`  
- 超出长度 → `400`，`code: "INPUT_TOO_LONG"`  
- 判定为 **纯链接输入策略触发** → `422`，`code: "LINK_INPUT_NOT_SUPPORTED"`（见 **§9**）

### 6.2 成功响应 `AnalyzeResponse`（HTTP 200）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `ok` | `true` | 是 | 成功固定为 `true` |
| `data` | `ParseResultPayload` | 是 | 与 **`ParseResultPreview`** 对齐的载荷（见 **§7**） |
| `meta` | `object` | 否 | 如 `model`（展示名）、`latencyMs`；**禁止**返回密钥或完整原始 HTTP |

### 6.3 错误响应（HTTP 4xx / 5xx）

统一 envelope（示例）：

```json
{
  "ok": false,
  "error": {
    "code": "MODEL_TIMEOUT",
    "message": "解析超时，请稍后重试。"
  }
}
```

**建议 `code` 枚举**（可删减，不可滥用为业务扩展）：

| `code` | HTTP | 含义 |
|--------|------|------|
| `EMPTY_INPUT` | 400 | 空输入 |
| `INPUT_TOO_LONG` | 400 | 超长 |
| `LINK_INPUT_NOT_SUPPORTED` | 422 | 主要为 URL，当前版本不支持抓取 |
| `MODEL_TIMEOUT` | 504 或 408 | 上游超时（实现择一并统一） |
| `MODEL_REFUSAL` | 422 | 模型拒答 / 内容政策 |
| `MODEL_ERROR` | 502 | 上游返回错误 |
| `INTERNAL_ERROR` | 500 | 未分类服务端错误 |

**说明**：HTTP 状态码与 `code` 的映射在开发时在 README 或本文档 **修订一节** 固化；**验收以 `code` + 用户可见文案为准**。

## 7. TypeScript 数据结构草案

> 与仓库 **`src/types/index.ts`** 中 **`ParseResultPreview`**、`ActionItem`、`KnowledgeCard` 对齐；若开发时发现字段冗余，**仅允许**在 **不破坏 1.0 页面结构** 前提下做小步重构。

### 7.1 API 专用请求 / 响应类型（草案）

```ts
/** POST /api/analyze 请求体 */
export interface AnalyzeRequestBody {
  text: string;
  locale?: string;
  requestId?: string;
}

/** 与 ParseResultPreview 对齐的成功载荷（服务端可填默认值） */
export interface ParseResultPayload {
  title: string;
  sourceLabel: string;
  sourceName?: string;
  tags?: string[];
  createdAtDisplay?: string;
  wordCount?: number;
  summary: string;
  keyInsights: string[];
  actionItems: ActionItemPayload[];
  knowledgeCards: KnowledgeCardPayload[];
}

export interface ActionItemPayload {
  id: string;
  content: string;
  description?: string;
  isDone: boolean;
}

export interface KnowledgeCardPayload {
  id: string;
  title: string;
  content: string;
  tag?: string;
}

export interface AnalyzeSuccessResponse {
  ok: true;
  data: ParseResultPayload;
  meta?: {
    model?: string;
    latencyMs?: number;
  };
}

export interface AnalyzeErrorResponse {
  ok: false;
  error: {
    code: string;
    message: string;
  };
}
```

### 7.2 字段与 UI 映射原则

- **`title`**：由模型从文本提炼；缺省时可用「用户文本首行截断」服务端兜底。  
- **`sourceLabel` / `sourceName`**：无外链抓取时，建议 **`sourceName` 为「用户笔记」或「粘贴正文」**，`sourceLabel` 与之同步，避免伪造外链来源。  
- **`wordCount`**：可对 **输入文本** 统计字数填入；与模型无关。  
- **`actionItems` / `knowledgeCards`**：必须带 **`id`**（客户端可用 `crypto.randomUUID()` 由服务端生成或在映射层补齐），以便列表渲染与 checkbox 状态。

## 8. Prompt 草案

> 以下为 **系统 + 用户** 消息草案；实际以供应商消息格式为准（OpenAI / Anthropic 等）。输出要求 **仅 JSON**，便于解析与映射。

### 8.1 系统指令（System）

```text
你是 Read2Action 的「阅读 → 结构化笔记」助手。用户会提供一段中文或外文纯文本（可能是文章节选、笔记、会议纪要等）。
你的任务：只基于用户提供的文本进行理解与提炼，不要编造文本中不存在的事实，不要输出外链或「请访问某 URL」类抓取指令。

输出要求：
1. 只输出一个合法的 JSON 对象，不要 Markdown 围栏，不要前后解释性文字。
2. JSON 必须符合以下键名与类型约定：
   - title: string，简短标题
   - summary: string，一句话总结（1～3 句，中文优先若用户文本主要为中文）
   - keyInsights: string[]，3～7 条，每条一句
   - actionItems: { content: string }[]，3～7 条可执行项，不要重复空洞措辞
   - knowledgeCards: { title: string, content: string, tag?: string }[]，2～5 张卡片；content 为对该概念的简短定义或提炼
3. 若文本过短或信息不足，仍返回 JSON，但在 summary 中诚实说明「信息有限」，并减少列表项数量。
4. 不要输出任何需要「联网抓取」才能完成的内容。
```

### 8.2 用户消息（User）

```text
以下为用户粘贴的纯文本，请按要求输出 JSON：

---
{{TEXT}}
---
```

### 8.3 解析与安全

- 服务端对模型输出做 **`JSON.parse`**；失败时走 **`MODEL_ERROR`/`MODEL_REFUSAL`** 与 **§10** 降级。  
- **可选**：JSON Schema / `response_format`（若供应商支持）以降低解析失败率。

## 9. 链接输入处理策略

### 9.1 目标

- **避免用户误以为 1.1 已支持自动读取链接正文**（`02` §8.4）。  
- **控制工程与合规风险**：不做抓取、不存第三方全文。

### 9.2 检测启发式（草案，可调整阈值）

**「主要为链接」** 满足以下 **任一** 时可触发策略（服务端与首页逻辑保持一致）：

- `trim` 后文本 **匹配** 单一 URL 模式（如 `^https?://\S+$`）且长度 &lt; 2048。  
- 文本中非空白字符中 **URL 字符占比** 过高（如 &gt; 85%）且行数 ≤ 3。

### 9.3 产品行为

- **推荐默认**：**阻止提交解析**，展示 **Toast 或输入区旁说明**，文案口径建议（与 `03` 一致）：  
  **「当前版本暂不支持自动读取链接正文，请复制正文后粘贴。」**  
- **不允许**：自动请求该 URL、服务端下载网页、嵌入 iframe 抓取等。

## 10. 错误与降级策略

| 场景 | 用户可见行为 | 技术处理要点 |
|------|----------------|----------------|
| 空输入 / 超长 | Toast 或内联提示；不发起请求 | 前端校验 + 服务端双检 |
| 链接形态 | 见 **§9** | 返回 `LINK_INPUT_NOT_SUPPORTED` |
| 模型超时 | 「解析超时，请稍后重试」 | 设置严格超时；可选一次重试（立项时决定是否做） |
| 模型拒答 | 「内容无法解析，请修改正文后重试」 | 映射供应商安全类错误 |
| JSON 解析失败 | 「解析结果格式异常，请重试」 | 记录 `requestId`；不暴露堆栈给用户 |
| 服务端 500 | 「服务暂时不可用」 | 泛化文案；详情仅服务端日志 |

**降级（不允许静默失败）**：

- **禁止**在失败时静默回退到 **「看起来像真 AI」的 mock 固定文案** 冒充成功（违反诚实边界与 `07` 叙事原则）。  
- **允许**：失败态下提供 **「返回首页」**、**「重试」**；重试即重新调用 **`/api/analyze`**。

**原文对照 / 方法论占位**：

- 详情页 **原文对照** 可展示 **用户本次粘贴的原文**（不涉及抓取则 **属于** 与 1.1 兼容的增强，但须在立项中单独勾选，避免与「占位未实现」混淆）。**不在本文强制纳入 1.1 首包**。

## 11. Cursor 后续执行原则

开发与改代码阶段，Agent **必须**遵守：

1. **范围**：只实现 **§2**；**严禁**引入 **§3** 所列能力；不接 Supabase / OAuth / 爬虫依赖。  
2. **架构**：不改六页路由与信息架构；**不大改** 视觉与全局 token（除非 P0 缺陷修复且与产品确认）。  
3. **密钥**：API Key **仅**服务端环境变量；不写入客户端 bundle、不提交到 Git。  
4. **小步**：每次变更说明 **修改文件列表**、**如何手测**、**遗留问题**（延续 `03` §6、`05` 习惯）。  
5. **契约优先**：若接口变更，**先**更新本文或 `03` 附件，**再**改代码。

## 13. 风险控制

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| **Scope creep** | 真实 AI 接入后易「顺便抓取 / 落库 / 登录」 | **§3** 写死 + Code Review 对照 **`04` 2026-05-16**；里程碑只验收 **§12** |
| **密钥泄露** | 客户端或日志打印密钥 | 仅服务端读取；日志脱敏；最小权限 Key |
| **成本失控** | 长文本、重复重试 | 输入 **max length**；IP / 简单 rate limit（无用户体系下的折中）；监控 token |
| **合规与内容安全** | 用户粘贴敏感或违法内容 | 不落库；可选供应商 **安全分类**；拒答映射 **§10**；隐私说明放 README |
| **解析失败率高** | 模型输出非 JSON | **§8.3** Schema / 重试 Prompt；失败走 **§10**，不冒充成功 |
| **过度承诺** | 对外宣传成「全能 AI 知识库」 | 坚持 **`02`/`07`** 诚实边界叙事 |
