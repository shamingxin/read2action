# Read2Action · 08 · 1.2「解析记录暂存与归档体验优化」启动方案

## 0. 执行摘要 / 最终执行口径

1. **1.2 只做本地记录生命周期补全**：围绕「自动暂存、项目归档、状态可见、空状态真实化」完成体验闭环，不扩展 1.3+ 能力。
2. **解析后不丢**：全局入口解析成功后，结果自动写入本地为未归档记录（`savedStatus: "temporary"`），可在最近列表找回并后续归档。
3. **项目内自动归档**：项目页内解析成功后，结果直接以已保存状态归属当前项目（`savedStatus: "saved"` + `projectId`）。
4. **真实空状态替代假数据**：最近列表、项目相关列表首次进入无本地数据时展示空状态，不默认展示 mock 假记录。
5. **冻结边界**：不接账号、不接数据库、不做多端同步，不重写 `/api/analyze`，不破坏 `/parsing` 防重复请求机制（A+B+C 继续保留）。

## 1. 版本定位与目标

- **版本定位**：在 1.1 已接入真实 AI 解析的基础上，补齐「本地解析记录不会丢」的产品体验。
- **核心目标**：
  - 统一真实空状态，去除默认 mock 污染。
  - 全局解析自动暂存到最近列表。
  - 项目内解析自动归档到当前项目。
  - 最近列表与结果页可区分未归档/已保存状态。
  - 修复主链路交互问题（状态、反馈、路由、文案、hydration 异常等）。
  - 明确提示本地保存边界（当前浏览器 localStorage）。

## 2. 范围内与范围外

### 2.1 范围内（1.2 必做）

1. 最近列表、项目列表、项目内笔记列表切换为真实空状态呈现。
2. 首页/侧栏「新笔记」解析成功后自动暂存为未归档记录。
3. 从最近列表或详情页可将未归档记录保存到项目。
4. 项目页顶部解析条解析成功后自动归档到当前项目。
5. 最近列表展示状态标签（「未归档」「已保存」）并支持直接归档操作。
6. 结果页根据解析来源展示差异化状态提示（自动暂存/已保存到项目）。
7. 在结果页或设置处展示本地保存说明文案。
8. 修复 PRD 定义的主链路交互 bug。

### 2.2 范围外（1.2 禁做）

1. 不接数据库，不做服务端持久化。
2. 不做登录注册，不做多端同步。
3. 不做文件上传解析，不做链接抓取。
4. 不做多模型 Provider 架构（保持单 DeepSeek 路线）。
5. 不做六页结构重构与大视觉改版。
6. 不做方法论自动生成。
7. **不重写 `/api/analyze`**。
8. **不改动 `/parsing` 防重复请求机制（A+B+C）**。

## 3. 核心流程定义

### 3.1 全局解析（首页/侧栏）

`输入文本 -> /parsing -> POST /api/analyze -> 成功后本地自动暂存(temporary/global) -> /result 显示“已自动暂存” -> 可选保存到项目 -> 最近列表可见`

### 3.2 项目内解析（项目页顶部）

`项目页输入文本 -> /parsing -> POST /api/analyze(携带 projectId) -> 成功后本地自动归档(saved/project) -> /result 显示“已保存到项目” -> 项目列表与最近列表可见`

### 3.3 空状态流程

`首次进入且无 localStorage 数据 -> 最近列表空状态 -> 项目内笔记列表空状态 -> 不注入 mock 假记录`

## 4. 数据与状态口径

在现有 `Note` 基础上补充并统一以下字段语义：

- `sourceContext: "global" | "project"`：解析发起来源。
- `savedStatus: "temporary" | "saved"`：归档状态。
- `projectId?: string`：项目归属（未归档可为空）。
- `createdAt` / `updatedAt`：排序与展示时间基线。

关键组合规则：

1. `temporary + global + undefined projectId`：全局解析未归档。
2. `saved + global + projectId`：全局解析后手动归档。
3. `saved + project + projectId`：项目内解析自动归档。

最近列表规则：

- 展示未归档记录与最近已保存记录。
- 以 `updatedAt` 倒序。
- 首次无数据必须显示空状态。

mock 策略：

- mock 仅保留为开发/演示参考，不默认进入产品 UI。
- 如需临时演示可通过开关启用，不影响正式行为口径。

## 5. 交互与缺陷修复口径

1. 保存后按钮与状态必须即时同步，避免 Toast 后状态不一致。
2. 所有关键按钮补齐 loading/disabled 反馈，杜绝“点击无响应”感知。
3. 解决刷新后 mock 混入、hydration 闪态导致的状态异常。
4. 本地记录详情页路由命中失败问题需修复，不得错误 `notFound`。
5. Toast 文案按来源区分，避免「全局暂存」与「项目保存」语义混淆。
6. 所有无数据场景提供明确引导，不出现空白界面。

## 6. 验收标准（DoD）

### 6.1 功能验收

- 清除 localStorage 后进入应用：最近列表与项目笔记列表均为真实空状态，无 mock。
- 首页解析成功后，不手动保存也可在最近列表找回记录。
- 刷新后暂存与已保存记录均可保留并正确展示状态。
- 未归档记录可成功保存到项目并状态切换为已保存。
- 项目页发起解析后，记录自动归属当前项目并可即时看到。
- 最近列表准确区分并展示「未归档」「已保存」标签。
- 本地暂存与已保存记录详情页均可正常打开。

### 6.2 稳定性与质量验收

- 单次解析仅产生 1 条 `POST /api/analyze` 请求（防重复机制保持有效）。
- API Key 仍仅在服务端读取，不下发前端。
- `npm run test`、`npm run lint`、`npm run build` 通过。
- 线上环境完成同口径复验。

## 7. 实施顺序建议

1. **阶段一：状态与空态基线**  
   先完成 mock 默认下线、空状态统一与 hydration 状态稳定，建立“真实无数据”基础。
2. **阶段二：记录生命周期补全**  
   落地自动暂存、自动归档、后续归档入口与状态标签展示。
3. **阶段三：交互修复与文案校准**  
   处理按钮反馈、Toast 语义、详情路由等主链路缺陷。
4. **阶段四：验收与回归**  
   按 DoD 全链路回归，重点验证“请求单发、刷新不丢、状态一致”。

## 8. 风险与控制

- **边界漂移风险**：实现中容易顺手扩展账号/落库；通过范围外清单与 Code Review 严控。
- **状态一致性风险**：暂存/归档状态在刷新与路由切换中可能错乱；以统一字段规则和渲染优先级收敛。
- **假数据回流风险**：历史 mock 逻辑可能在特定初始化路径重入；以“默认不注入”作为硬约束。
- **认知偏差风险**：用户误解为云端同步；在结果页/设置处明确本地保存边界文案。

## 9. Cursor 执行约束（启动期）

1. 严格按本方案与 1.2 PRD 执行，不新增需求。
2. 修改优先最小化，避免重构稳定链路。
3. 不重写 `/api/analyze`，不破坏 `/parsing` 现有防重复机制。
4. 每次提交需附：修改文件清单、手测路径、剩余风险。
