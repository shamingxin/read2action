## 2. 技术方向确认

| 层级 | 选型 | 说明 |
|------|------|------|
| 前端 / SSR | **Next.js** App Router | Auth 会话用 Supabase 官方 SSR 模式（`@supabase/ssr`，**1.3-C** 安装） |
| 认证 | **Supabase Auth** | 1.3 最小：**Email + Password**；OAuth 列为范围外 |
| 数据库 | **Supabase PostgreSQL** | 业务表 `public`；`auth.users` 托管 |
| 部署 | **Vercel** | 延续 `read2action.vercel.app` |
| AI | **现有 `POST /api/analyze`** | **不重写** A+B+C；`AI_*` 仅服务端 |

**三条铁律**

1. **RLS 是唯一云端隔离手段** — anon key 可进前端，但 **不能** 靠藏 key 保安全。  
2. **未登录 = 1.2 完整体验** — localStorage 主链路不变。  
3. **最小可落地** — 先三表跑通，暂缓非阻塞表与复杂双写。

## 3. 账号体系规划

### 3.1 产品原则：不强制注册

| 原则 | 说明 |
|------|------|
| **默认可用** | 游客可直接：输入文本 → 解析 → `/result` → localStorage 暂存 / 保存到项目 |
| **登录是增值** | 注册 / 登录为 **可选**；价值主张是 **换设备找回、云端备份** |
| **无硬拦截** | 首页、解析、结果页 **不得** 弹「必须登录」；登录入口为侧栏 / 顶栏 **弱引导** |
| **登出不删本地** | 登出后回到游客模式，**保留** `r2a:localSavedNotes` |

### 3.2 注册

- **方式**：Email + Password（`signUp`）。  
- **邮箱验证**：Production 建议开启 Confirm email；本地 / Preview 可关以便验收。  
- **注册后数据**：`auth.users` 创建 → **Trigger** 自动插入 `profiles`（`id = auth.users.id`）。  
- **UI**：`/signup` 或登录页 Tab；遵循 paper-calm 设计规范（**1.3-C**）。  
- **不做**：手机号、OAuth、邀请码、团队空间（均属 1.3 范围外）。

### 3.3 登录

- **方式**：Email + Password（`signInWithPassword`）。  
- **入口**：侧栏 / 顶栏「登录」。  
- **成功后续**（**1.3-E** 实现迁移检测）：若本地有未迁数据 → 走 §8 迁移引导；否则直接进入云端模式。  
- **失败态**：友好文案（密码错误、未验证邮箱、网络失败）；**不**向用户展示 Supabase 原始 error code。

### 3.4 登出

- 调用 `signOut`，清除 Auth Cookie。  
- UI 回 **未登录本地模式**（§3.6）。  
- **禁止**：登出时 `localStorage.removeItem('r2a:localSavedNotes')` 或静默清本地数据。

### 3.5 登录态保持

- **机制**：Supabase JWT + Refresh Token；Next.js **Middleware** 刷新会话（`@supabase/ssr` 推荐流）。  
- **读取**：Server Component / Route Handler 用 `createServerClient` 取 `auth.getUser()`；Client Component 用 browser client。  
- **无会话时**：**不查询** Supabase 业务表；页面逻辑与 **1.2 一致**（local ∪ mock）。  
- **1.3 不做**：自定义 session 表、多端踢下线、Remember me 单独实现。

### 3.6 未登录本地模式（游客）

与 **1.2 冻结行为** 对齐，作为 1.3 回归基线：

| 能力 | 行为 |
|------|------|
| 解析主链路 | `/` → `/parsing` → `POST /api/analyze`（单次）→ `/result` |
| 自动暂存 | `savedStatus: temporary`、`sourceContext: global` 写入 localStorage |
| 保存到项目 | 升级 `saved`、写入 `projectId` |
| 项目内解析 | 自动归档到当前 `projectId` |
| 列表 / 详情 | 读 `r2a:localSavedNotes` ∪ mock（`?mock=true`） |
| 云端 | **零访问** — 不发 Supabase 业务查询 |
| 换设备 | ❌ 数据不跟随 |

**本地数据契约**（只读引用，1.3-A 不改代码）：

- Key：`r2a:localSavedNotes`  
- 结构：`{ v: 1, notes: Note[] }`（`src/lib/local-saved-notes.ts`）  
- 未归档占位：`projectId = "_temporary"`（`R2A_TEMPORARY_PROJECT_ID`）

### 3.7 登录后云端模式（已认证）

| 能力 | 行为 |
|------|------|
| 解析主链路 | 与游客相同；**A+B+C 不变** |
| **读** 项目 / 笔记列表 / 详情 | **优先 Supabase**（`projects` / `notes`，RLS 过滤） |
| **写** 新建项目、保存笔记、编辑 | **写 Supabase**；1.3-D 起 **单写云端**（不做长期双写） |
| 侧栏「最近」 | 读云端 `notes`，按 `updated_at` 降序；**1.3-E 前** 可与未迁 local 合并（见 §8.6） |
| URL / ID | 新业务用 **云端 UUID**；迁移保留 `local_id` 映射，不强迫 URL 继续用 `sha` 等字符串 |
| 换设备 | ✅ 登录同一账号可拉回云端数据 |
| localStorage | **迁移完成前不删**；迁移后作只读备份，直至用户主动清理或后续版本决策 |

### 3.8 双模式对照总表

| 维度 | 未登录（游客） | 已登录（云端） |
|------|----------------|----------------|
| 能否解析 | ✅ | ✅ |
| 数据主存储 | localStorage | Supabase |
| 侧栏最近 / 项目列表数据源 | local ∪ mock | 云端（+ 过渡期 local 合并） |
| 换设备同步 | ❌ | ✅ |
| 访问他人云端数据 | ❌ | ❌（RLS） |
| 是否必须注册 | **否** | 用户主动选择 |

## 4. 数据库结构规划（最小可落地方案）

### 4.1 1.3 必建三张表

**1.3-B** 首次 migration **只创建**：

```text
auth.users (Supabase 托管)
    │
    ├── 1:1 ── profiles
    │
    ├── 1:N ── projects
    │              │
    │              └── 1:N ── notes
```

| 表 | 职责 |
|----|------|
| **`profiles`** | 用户资料 + **迁移状态**（替代独立迁移日志表的最小方案） |
| **`projects`** | 用户项目；`local_id` 映射原 local `projectId` |
| **`notes`** | 笔记全量字段；`local_id` 映射原 local `note.id`，**防重复迁移** |

> `auth.users` 不直接写入业务字段；所有业务行带 `user_id uuid NOT NULL`。

### 4.2 暂缓项（1.3 最小闭环不建表）

| 表 / 能力 | 决策 | 理由 |
|-----------|------|------|
| **`analysis_results`** | **暂缓** | 解析结果已落在 `notes`（`summary`、`key_insights`、`raw_content` 等）；`/api/analyze` 仍走 session → 保存进 `notes`。若未来要「未保存解析历史」再单独立项 |
| **`local_migration_logs`** | **暂缓** | 迁移态用 `profiles.migration_status` + `migration_completed_at` + 可选 localStorage 标记 `r2a:migrationCompletedAt` 即可；审计需求强时再 **1.3-E+** 加表 |
| **软删除 / 回收站** | 不做 | 1.4+ 或体验专项 |
| **Storage 附件** | 不做 | 属 1.4 |

### 4.3 `profiles`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | `uuid` | PK, FK → `auth.users(id)` ON DELETE CASCADE | 与 Auth 用户同 id |
| `display_name` | `text` | nullable | 展示名 |
| `avatar_url` | `text` | nullable | 1.3 可空 |
| `migration_status` | `text` | NOT NULL, default `'none'` | 见下表 |
| `migration_completed_at` | `timestamptz` | nullable | 迁移成功时间 |
| `created_at` | `timestamptz` | NOT NULL, default `now()` | |
| `updated_at` | `timestamptz` | NOT NULL, default `now()` | |

**`migration_status` 枚举**

| 值 | 含义 |
|----|------|
| `none` | 新用户，尚未检测本地 |
| `pending_user_confirm` | 检测到本地数据，待用户确认 |
| `in_progress` | 迁移进行中 |
| `completed` | 迁移完成或无可迁数据 |
| `failed` | 迁移失败，可重试 |

### 4.4 `projects`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | `uuid` | PK, default `gen_random_uuid()` | 云端主键 |
| `user_id` | `uuid` | NOT NULL, FK → `auth.users(id)` ON DELETE CASCADE | RLS 键 |
| `name` | `text` | NOT NULL | |
| `description` | `text` | nullable | |
| `local_id` | `text` | nullable | 原 local `projectId`（如 `sha`、`_temporary`） |
| `created_at` | `timestamptz` | NOT NULL | 迁移时保留原时间 |
| `updated_at` | `timestamptz` | NOT NULL | |

**约束与索引**

- `UNIQUE (user_id, local_id)` WHERE `local_id IS NOT NULL` — 同用户下本地项目只映射一行。  
- `INDEX (user_id, updated_at DESC)` — 列表排序。

**系统项目**：迁移时若存在 `projectId = '_temporary'`，为该用户 upsert 一条 `local_id = '_temporary'` 的项目（名称如「未归档」）。

### 4.5 `notes`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | `uuid` | PK, default `gen_random_uuid()` | 云端主键 |
| `user_id` | `uuid` | NOT NULL, FK → `auth.users(id)` ON DELETE CASCADE | |
| `project_id` | `uuid` | NOT NULL, FK → `projects(id)` ON DELETE CASCADE | |
| `title` | `text` | NOT NULL | |
| `source_type` | `text` | NOT NULL | 对齐 `SourceType` |
| `source_name` | `text` | nullable | |
| `source_url` | `text` | nullable | |
| `raw_content` | `text` | NOT NULL, default `''` | 详情加载；列表查询 **不 SELECT** |
| `summary` | `text` | NOT NULL, default `''` | |
| `key_insights` | `jsonb` | NOT NULL, default `'[]'` | |
| `action_items` | `jsonb` | NOT NULL, default `'[]'` | |
| `knowledge_cards` | `jsonb` | NOT NULL, default `'[]'` | |
| `tags` | `jsonb` | NOT NULL, default `'[]'` | |
| `word_count` | `integer` | NOT NULL, default `0` | |
| `saved_status` | `text` | NOT NULL, default `'saved'` | `temporary` \| `saved` |
| `source_context` | `text` | NOT NULL, default `'global'` | `global` \| `project` |
| **`local_id`** | `text` | nullable | **原 localStorage `note.id`；迁移幂等键** |
| `created_at` | `timestamptz` | NOT NULL | |
| `updated_at` | `timestamptz` | NOT NULL | |

**约束与索引**

- **`UNIQUE (user_id, local_id)`** WHERE `local_id IS NOT NULL` — **防重复迁移核心**；已存在则 **跳过**（不覆盖云端，见 §8.4）。  
- `INDEX (user_id, project_id, updated_at DESC)` — 项目内列表。

### 4.6 `local_id` 防重复迁移（设计要点）

```text
localStorage note.id  ──►  notes.local_id
localStorage projectId ──►  projects.local_id  ──►  notes.project_id (UUID FK)
```

1. 迁移前查 `notes` 是否已有 `(user_id, local_id)` → **有则 skip**。  
2. 迁移前查 `projects` 是否已有 `(user_id, local_id)` → **有则复用**该 `projects.id`。  
3. 云端 **新建**数据（非迁移）：`local_id = NULL`，仅用 UUID。  
4. **禁止**用 `notes.id` 复用 local 字符串 id 作主键。

### 4.7 与 `src/types` 映射

| TypeScript | 表 / 列 |
|------------|---------|
| `Project.id` | `projects.id`（新）或经 `local_id` 查表 |
| `Project.name` | `projects.name` |
| `Note.*` | `notes` 同名列（camelCase → snake_case） |
| `ParseResultPreview` | 保存时写入 `notes`，**不**单独建 `analysis_results` |

## 5. 权限策略（RLS）

### 5.1 总原则（必须遵守）

| # | 规则 |
|---|------|
| 1 | **`profiles`、`projects`、`notes` 全部 `ENABLE ROW LEVEL SECURITY`** |
| 2 | 所有业务策略条件：**`user_id = auth.uid()`** 或 **`id = auth.uid()`**（profiles） |
| 3 | 用户 **只能读写自己的** 项目与笔记 |
| 4 | **未登录**（`auth.uid()` 为 null）→ 业务表 **SELECT/INSERT/UPDATE/DELETE 均无允许策略** → 等价于不能访问云端数据 |
| 5 | **`NEXT_PUBLIC_SUPABASE_ANON_KEY` 可在浏览器使用**，安全靠 **RLS**，不靠隐藏 anon key |
| 6 | **`SUPABASE_SERVICE_ROLE_KEY` 仅服务端**（Route Handler / Server Action / 受控脚本）；**禁止** `NEXT_PUBLIC_` 前缀；**禁止**进 Client Component / 浏览器 bundle |
| 7 | **`AI_API_KEY`（DeepSeek）仅服务端**；与 1.1 相同，仅 `process.env` 于 `/api/analyze` |

### 5.2 分表策略（1.3-B SQL 依据）

**`profiles`**

| 操作 | 策略 |
|------|------|
| SELECT | `auth.uid() = id` |
| UPDATE | `auth.uid() = id` |
| INSERT | Trigger 在 `auth.users` 创建后 **service role** 插入；或 `auth.uid() = id` 的受限 INSERT |

**`projects` / `notes`**

| 操作 | 策略 |
|------|------|
| SELECT | `user_id = auth.uid()` |
| INSERT | `user_id = auth.uid()`（**WITH CHECK** 同条件，防伪造 `user_id`） |
| UPDATE | `user_id = auth.uid()` |
| DELETE | 1.3 **可不开放**；若开放则 `user_id = auth.uid()` |

**默认拒绝**：未单独 `GRANT` 给 `anon` / `authenticated` 的 broad 权限；仅通过 RLS policy 放行。

### 5.3 应用层纪律

- 无 session 的 RSC / 页面：**不 import** 云端列表查询。  
- **禁止** 在客户端使用 service role 拉全表。  
- 迁移 Route Handler：优先 **用户 JWT + RLS** 逐条 insert；若用 service role，必须 **校验 body 内 user_id 与 JWT 一致** 且接口 **鉴权**。

### 5.4 实现后安全验收（1.3-C / 1.3-D 必做）

- [ ] 用户 A 的 access token **无法** SELECT 用户 B 的 `notes` / `projects`  
- [ ] 未登录直接调 Supabase REST **读不到** 业务行  
- [ ] INSERT 时伪造 `user_id = B` **被拒绝**  
- [ ] 构建产物 / 浏览器 Sources **无** `service_role` 字符串  
- [ ] `AI_API_KEY` 不出现在 client bundle

## 6. localStorage → Supabase 迁移方案

### 6.1 范围

| 迁移 | 不迁移 |
|------|--------|
| `r2a:localSavedNotes` 内 `notes[]` | mock 数据（`?mock=true`） |
| 涉及的 `projectId`（含 `_temporary`） | sessionStorage 解析中间态 |
| | 其他 localStorage key |

### 6.2 流程总览

```text
登录成功
  → 读 profiles.migration_status
  → 若 completed：跳过引导
  → 若 none：读 localStorage
       → 无数据：migration_status = completed，结束
       → 有数据：migration_status = pending_user_confirm，弹窗
            → 用户「稍后再说」：保持 pending，不迁
            → 用户「立即导入」：in_progress → 服务端幂等导入 → completed | failed
```

### 6.3 首次登录检测本地数据

1. **触发**：`signIn` / `signUp` 成功后，**1.3-E** 客户端读 `readLocalSavedNotesEnvelope()`。  
2. **条件**：`profiles.migration_status = 'none'` 且 `notes.length > 0`。  
3. **动作**：更新 `migration_status = 'pending_user_confirm'`；展示非阻断 Modal（条数、项目数估算）。  
4. **无本地数据**：直接 `migration_status = 'completed'`，**不弹窗**。

### 6.4 用户确认后迁移

1. 用户点击「立即导入」→ `migration_status = 'in_progress'`。  
2. **服务端**（Route Handler，**1.3-E**）按条处理：  
   - 按 `note.projectId` 分组 → upsert `projects`（`local_id` 幂等）。  
   - `_temporary` → 确保存在 `local_id='_temporary'` 的项目。  
   - 对每条 note：若已存在 `(user_id, local_id)` → **跳过**；否则 INSERT `notes`。  
3. 全部成功 → `migration_status = 'completed'`，`migration_completed_at = now()`。  
4. 可选：写入 `localStorage` 标记 `r2a:migrationCompletedAt`（便于客户端快速判断，**非**删数据）。

### 6.5 防重复迁移

| 层级 | 机制 |
|------|------|
| **DB** | `UNIQUE (user_id, local_id)` on `notes` / `projects` |
| **导入逻辑** | 插入前 `SELECT` 已存在 → **skip**，不 UPDATE 覆盖（避免云端新数据被旧 local 覆盖） |
| **状态机** | `migration_status = 'completed'` 后 **不再自动** 全量迁移弹窗 |
| **重试** | 仅 `failed` 或用户主动「重试迁移」时再次执行；仍 skip 已有 `local_id` |

### 6.6 迁移失败：保留本地数据

- 任一步失败 → `migration_status = 'failed'`；**不** `removeItem('r2a:localSavedNotes')`。  
- UI：错误提示 +「重试」；本地列表 **仍可正常使用**（游客能力不退化）。  
- 部分成功：已写入的 `local_id` 在重试时 skip，从未成功的条继续导入。

### 6.7 不静默删除本地数据

- 1.3 **默认策略**：迁移成功后 **不自动清除** localStorage。  
- **禁止**：迁移成功 / 登录成功时静默 `clear()` 或覆盖 local。  
- 若未来提供「清除本地副本」：必须 **二次确认**（非 1.3-E 范围）。

### 6.8 避免本地 / 云端状态混乱

**分阶段读合并规则**（写入 1.3-D / 1.3-E 启动方案）：

| 阶段 | 已登录列表数据源 | 写 |
|------|------------------|-----|
| **1.3-D**（未做迁移） | 云端为主；**未** `completed` 时可合并 local 中 `local_id` 未出现在云端的条 | **仅写云端** |
| **1.3-E**（迁移中） | 同上，避免双弹窗 | 仅写云端 |
| **迁移 completed** | **仅云端**（local 不再并入列表，防重复展示） | 仅写云端 |
| **登出** | 回 local ∪ mock | 仅写 local |

**去重键**：合并展示时以 `notes.local_id === localNote.id` 判重，**云端优先**。

**URL**：迁移后新打开详情用云端 `notes.id`（UUID）；旧书签 local id 可在 **1.3-E** 用 `local_id` 查询重定向（可选，不阻塞最小闭环）。

## 7. 环境变量规划

### 7.1 变量清单

| 变量名 | 运行位置 | 是否可前端 | 说明 |
|--------|----------|------------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | 客户端 + 服务端 | **是**（故意公开） | Supabase 项目 URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | 客户端 + 服务端 | **是**（故意公开） | 匿名 key；**必须**配合 RLS |
| `SUPABASE_SERVICE_ROLE_KEY` | **仅服务端** | **否** | 绕过 RLS；迁移/Trigger 等受控场景 |
| `AI_API_KEY` | **仅服务端** | **否** | DeepSeek / 兼容 API |
| `AI_BASE_URL` | **仅服务端** | **否** | 可选 |
| `AI_MODEL` | **仅服务端** | **否** | 可选 |

### 7.2 `NEXT_PUBLIC_` 纪律

- Next.js 会把 `NEXT_PUBLIC_*` **打进浏览器 bundle**，任何用户可在 DevTools 看到。  
- **可放**：Supabase URL、anon key（设计如此）。  
- **绝不可放**：`SUPABASE_SERVICE_ROLE_KEY`、`AI_API_KEY`、任何 Admin / 私有密钥。  
- **错误示例**：`NEXT_PUBLIC_AI_API_KEY` — **禁止**。

### 7.3 Vercel / 本地

| 环境 | 配置 |
|------|------|
| **Production** | 全部服务端变量 + `NEXT_PUBLIC_SUPABASE_*` |
| **Preview** | 建议独立 Supabase 项目或同项目不同 Auth 策略 |
| **Development** | `.env.local` 手工维护；**不入库** |

### 7.4 更新时机

- **1.3-A**：不改 `.env.example`。  
- **1.3-B**：更新 `.env.example` 增加 Supabase 占位项；Vercel 控制台配置；本地 `.env.local` 由开发者自行填写。

## 9. 风险清单

| 风险 | 影响 | 缓解 |
|------|------|------|
| **RLS 配置错误 → 数据串用户** | A 看见 B 的笔记 | 默认拒绝；`ENABLE RLS` 三张表；双账号验收；Policy 单元测试 |
| **service_role key 暴露** | 全库可被爬 | 仅服务端 env；CI 扫描 bundle；禁止 `NEXT_PUBLIC_` |
| **迁移重复导入** | 列表重复条 | `UNIQUE (user_id, local_id)` + 导入前 skip |
| **迁移失败 → 误以为丢数据** | 用户恐慌 | 失败不删 local；明确 UI；`failed` 可重试 |
| **未登录 / 已登录状态混乱** | 双份列表、闪动 | §6.8 分阶段读规则；`completed` 后不合 local |
| **本地与云端口径不一致** | 同条笔记内容不同 | 合并时云端优先；skip 已有 `local_id` 不覆盖 |
| **过早复杂化** | 1.3 无法收口 | 三表优先；暂缓 `analysis_results` / `local_migration_logs`；单写云端 |
| **Vercel 漏配 env** | 线上 Auth 全挂 | 1.3-B checklist；部署后 smoke test |
| **破坏 1.2 解析链路** | 费用 / 回归 | 1.3-B/C 不改 `/api/analyze`；保留 A+B+C |
| **UUID 与 local 路由不一致** | 404 | `local_id` 查询；新链用 UUID |
| **列表拉 `raw_content`** | 慢查询 | 列表省略大字段 |
