# Read2Action · 文档目录说明

本目录按 **四类** 组织，便于新对话 `@` 引用与版本接力：

```text
docs/
├── README.md
├── PRD/
├── 核心记录/
├── 版本启动方案/
└── 阶段复盘/
```

| 目录 | 用途 |
|------|------|
| **`PRD/`** | 各版本的 **产品定义 / 规划文档**。当前含 **1.0～1.2** 正式 PRD，及 **1.3/** 下分阶段方案（**1.3-A ✅ 已冻结**）。 |
| **`核心记录/`** | 长期维护的 **`01`～`07`** 七份中文核心文档，外加 **`Read2Action_协作流程SOP_v1.0.md`**（协作流程 SOP）。 |
| **`版本启动方案/`** | 各版本 **开工前** 的执行契约（含开发步骤、执行边界、文件范围、验收方式；当前含 **1.1** **`08_1.1…`**、**1.2** **`08_1.2…`**）。 |
| **`阶段复盘/`** | 各阶段 / 版本的 **复盘与验收事实归档**（如 1.0 一页式总结、**9.3 走查全文**、**1.1 P0 排障实录**、开发阶段分表、1.1 上线收尾、1.1 / 1.2 正式总复盘等）。 |

> **结构规范（2026-06-09 固化）**：Read2Action 后续 docs 采用「**PRD / 版本启动方案 / 阶段复盘 / 核心记录**」四类结构。PRD 记录版本计划，启动方案记录执行契约，阶段复盘记录实际过程与问题闭环，核心记录维护长期项目记忆与 AI 协作 SOP。后续版本只按既有目录新增或更新文档，**不新增平行目录**，**不大规模重命名历史文件**。详 **`docs/阶段复盘/Read2Action_1.2_解析记录暂存与归档体验优化版总复盘.md` §7**。

## docs 整理原则（PRD vs 核心记录 vs 版本启动方案 vs 阶段复盘）

1. **`PRD/`**：记录每个版本的 **产品定义**；内容包括版本定位、版本目标、用户价值、范围内、范围外、核心流程、数据/状态规则、验收标准、现实校准；**不写执行细节**，不直接替代 Cursor 开发指令。
2. **`核心记录/01`～`07`**：**保持精简**，服务 **新对话接力** 与 **项目推进**；只写 **当前状态、关键决策、下一步、边界与索引**；**不写**可独立成篇的长过程表；**不把大量复盘细节塞回** `01`～`07`。  
3. **`版本启动方案/`**：记录每个版本 **开工前的执行契约**；重点写 **开发步骤、执行边界、可能修改的文件范围、Cursor 执行规则、分阶段验收方式**（与 `08` 体例一致）；**每个版本开工前单独生成**。
4. **`阶段复盘/`**：允许 **更详细**；必须保留 **过程、问题、尝试路径、最终方案、工作流反思、作品集素材**；**禁止**为迁就 `02` 篇幅而 **过度压缩成一句结论**（可写摘要 + **链到本目录实录**）。

**默认接力**：优先 **`@docs/核心记录/02_当前进度快照.md`**，并配合 **`docs/核心记录/03_下一步任务清单.md`**、**`docs/核心记录/04_重要决策日志.md`**、**`docs/核心记录/05_Cursor执行规则.md`**。进入 **1.x 代码或契约细节** 时补充 **`docs/版本启动方案/`** 下 **`08`** 或未来 **`09`** 等文件。

**路径约定**：下文与索引中的 `docs/...` 路径均自 **仓库根目录** 起算。
