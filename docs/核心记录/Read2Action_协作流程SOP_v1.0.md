# Read2Action 协作流程 SOP

版本：v1.0  
适用阶段：Read2Action 1.1 及后续所有版本迭代  
用途：规范 ChatGPT、Cursor、GitHub、Vercel、项目文档之间的协作流程，避免长对话、文档混乱、任务失控。

## 0. 核心原则

Read2Action 后续迭代不再靠“复制多个项目文件夹”来管理版本，而是使用：

1. 一个本地项目文件夹
2. 一个 GitHub 仓库
3. 一套 docs 文档
4. 小步任务拆分
5. 每轮开发后更新文档与提交记录

## 2. 本地文件夹管理规则

### 2.1 是否需要为 1.1 / 1.2 / 1.3 新建文件夹？

不需要。

不要这样做：

```text
read2action-1.0
read2action-1.1
read2action-1.2
read2action-2.0
```

推荐只保留一个主项目文件夹：

```text
/Users/shasha/read2action
```

原因：

1. 后续版本是在同一个产品上持续迭代
2. GitHub 会记录每次代码变化
3. Vercel 会部署最新线上版本
4. 多文件夹容易导致不知道哪个是最新版本
5. Cursor、终端、GitHub 配置容易混乱

### 2.2 如果当前文件夹叫 read2action 1.0，可以改名吗？

可以改，但需要注意：

改名前：

1. 确认当前代码已经提交到 GitHub
2. 关闭 Cursor 当前窗口
3. 在 Finder 里把文件夹改名为 read2action
4. 重新用 Cursor 打开新的 read2action 文件夹
5. 在 Cursor 终端运行：

```bash
pwd
```

确认路径是：

```text
/Users/shasha/read2action
```

再运行：

```bash
git status
npm run lint
npm run build
```

如果都正常，就说明改名没有问题。

如果担心出错，也可以暂时不改名，等 1.1 开始前再改。文件夹名字不影响线上功能，但长期建议改成稳定名称。

## 3. GitHub / Vercel / 本地文件夹的关系

### 3.1 本地文件夹

本地文件夹是你实际开发代码的地方。

### 3.2 GitHub

GitHub 是代码版本记录和备份的地方。

每次提交后，GitHub 会记录：

- 改了哪些文件
- 谁改的
- 什么时候改的
- 改动说明是什么

所以不需要靠复制文件夹保存版本。

### 3.3 Vercel

Vercel 是线上部署的地方。

一般流程是：

```text
本地修改代码
↓
测试通过
↓
提交到 GitHub
↓
推送到 GitHub
↓
Vercel 自动部署
↓
线上版本更新
```

### 3.4 正确理解

不是“因为数据和功能都更新到线上了，所以不用建文件夹”。

更准确地说：

```text
代码版本由 GitHub 管理；
线上版本由 Vercel 部署；
本地只保留一个主项目文件夹；
产品版本通过文档和 Git commit 记录。
```

## 4. 文档体系规则

从 1.1 开始，Read2Action 使用“两层文档体系”。

---

## 4.1 长期稳定文档

这些文档不需要每轮都改，只在方向、规则、重要决策变化时更新。

### 01_项目总说明.md

用途：

- 项目背景
- 产品定位
- 核心用户
- 核心场景
- 核心流程
- 技术栈
- 页面结构
- 版本规划

### 04_重要决策日志.md

用途：

- 记录关键产品决策
- 记录关键技术决策
- 记录为什么这么做
- 记录当时放弃了什么方案

示例：

```text
2026-05-11
决策：1.1 只做文本总结，不做链接全文抓取。
原因：链接抓取涉及网页解析、权限、反爬、失败率，容易拖慢 MVP。
影响：1.1 先保证真实 AI 总结可用，链接能力放入 1.2 或 2.0。
```

### 05_Cursor执行规则.md

用途：

- 给 Cursor 的长期执行规则
- 规定哪些能改，哪些不能乱改
- 规定每次动手前必须先输出计划
- 规定每次开发后必须跑 lint / build
- 规定每次结束后必须更新文档

### 07_方法论沉淀.md

用途：

- 记录 AI 协作流程
- 记录设计师如何和 ChatGPT / Cursor 协作
- 记录问题、解决方案、可复用经验
- 后续用于作品集内容沉淀

## 5. 新开 ChatGPT 对话 SOP

### 5.1 什么时候需要新开 ChatGPT 对话？

建议在以下情况新开：

1. 当前对话很长，阅读变慢
2. 当前阶段已经结束
3. 要进入新阶段，例如从 1.0 复盘进入 1.1 开发
4. 需要避免上下文混乱
5. 已经积累了较多截图、代码反馈、判断过程

### 5.2 新开 ChatGPT 对话时发什么？

优先上传：

```text
02_当前进度快照.md
03_下一步任务清单.md
```

如果涉及大方向判断，再上传：

```text
01_项目总说明.md
04_重要决策日志.md
```

如果涉及 Cursor 规则，再上传：

```text
05_Cursor执行规则.md
```

不需要每次都上传全部文档。

### 5.3 新开 ChatGPT 对话话术

```text
卡卡，请接着 Read2Action 项目继续。

我已上传最新的：
1. 02_当前进度快照.md
2. 03_下一步任务清单.md

请先阅读文档，只围绕当前阶段继续，不要重新推翻已冻结的 1.0 产品方向、页面结构和视觉方向。

当前目标是：继续推进文档中记录的下一步任务。
如果你发现文档有冲突，请先指出冲突，不要直接让我进 Cursor 改代码。
```

## 6. 结束 ChatGPT 对话 SOP

### 6.1 不再要求每次生成一堆新 md

以后结束 ChatGPT 对话时，不要每次都让 ChatGPT 重新生成 7 个完整 md。

推荐只让 ChatGPT 输出“更新内容”，再交给 Cursor 写入 docs。

### 6.2 结束话术

```text
卡卡，请总结本轮内容，按以下结构输出：

1. 本轮结论
2. 需要更新到 02_当前进度快照.md 的内容
3. 需要更新到 03_下一步任务清单.md 的内容
4. 如有重要决策，更新到 04_重要决策日志.md
5. 如有方法论沉淀，更新到 07_方法论沉淀.md
6. 给 Cursor 的接力话术

先不要生成新的完整文档，只输出可复制的更新内容。
```

## 7. 新开 Cursor Agent SOP

### 7.1 什么时候需要新开 Cursor Agent？

建议在以下情况新开：

1. Cursor 当前上下文超过 70% - 80%
2. 一个阶段任务已经完成
3. Cursor 开始混淆需求
4. 要切换到一个新模块
5. 需要重新让 Cursor 严格读取 docs

### 7.2 新开 Cursor Agent 话术

```text
请先阅读 docs 目录里的所有 Read2Action 文档，尤其是：

1. 02_当前进度快照.md
2. 03_下一步任务清单.md
3. 05_Cursor执行规则.md

阅读后先不要改代码。

请先用中文总结：
1. 当前项目状态
2. 当前阶段目标
3. 本轮你准备做什么
4. 预计会修改哪些文件
5. 不会碰哪些内容
6. 需要我确认的问题

等我确认后再开始。
```

### 7.3 Cursor 动手前必须确认

Cursor 输出计划后，你需要检查：

1. 是否理解当前版本目标
2. 是否只做本轮任务
3. 是否乱加了新功能
4. 是否要改太多文件
5. 是否影响 1.0 已完成页面
6. 是否明确测试方式

如果不确定，把 Cursor 的计划截图或复制给 ChatGPT 判断。

## 8. Cursor 开发中 SOP

### 8.1 一次只做一个小任务

正确示例：

```text
只新增 /api/analyze，用于接收用户输入文本并返回 mock JSON，先不接真实模型。
```

```text
只把首页提交逻辑改成调用 /api/analyze，其他页面不要动。
```

```text
只修复结果页对真实 JSON 数据的渲染，不改视觉样式。
```

错误示例：

```text
帮我把 1.1 都做完。
```

```text
顺便把登录、数据库、链接抓取也做了。
```

```text
你看着优化一下所有页面。
```

### 8.2 Cursor 每轮开发后必须输出

```text
请总结本轮修改：

1. 修改了哪些文件
2. 每个文件改了什么
3. 如何测试
4. npm run lint 是否通过
5. npm run build 是否通过
6. 是否影响原有 1.0 页面
7. 是否需要更新 docs
8. 下一步建议做什么
```

## 9. 每次正式修改前的安全流程

每次开始一个新阶段或较大修改前，先让 Cursor 检查 Git 状态：

```bash
git status
```

如果当前有未提交内容，先提交或确认是否需要保存。

推荐提交命令：

```bash
git add .
git commit -m "docs: update project status before next iteration"
git push
```

对于功能开发提交，可以使用：

```bash
git add .
git commit -m "feat: add AI text analysis API"
git push
```

对于修复问题提交，可以使用：

```bash
git add .
git commit -m "fix: improve result page data rendering"
git push
```

对于文档更新提交，可以使用：

```bash
git add .
git commit -m "docs: update Read2Action 1.1 planning"
git push
```

## 13. 每轮协作的推荐节奏

### ChatGPT 阶段

用于：

- 定方向
- 拆任务
- 判断 Cursor 方案
- 复盘总结
- 生成交接话术
- 沉淀方法论

### Cursor 阶段

用于：

- 读 docs
- 改代码
- 跑检查
- 更新 docs
- 提交 GitHub

### 用户阶段

用户负责：

- 判断体验是否符合预期
- 截图反馈
- 控制范围
- 决定是否进入下一阶段
- 确认重要决策

## 14. 最重要的执行纪律

1. 不要一次让 Cursor 做太多
2. 不要每轮都新建完整 md
3. 不要为每个版本复制项目文件夹
4. 不要在 1.1 顺手做数据库、登录、链接抓取
5. 不要让 Cursor 未确认计划就直接改代码
6. 每个阶段结束都要更新当前进度和下一步任务
7. 每个重要决策都要写清楚原因
8. 每个可复用经验都要沉淀到方法论文档
9. 线上稳定版本优先，不追求一次做完
10. 先可用，再好用，再高级

## 15. 一句话总结

Read2Action 后续迭代采用：

```text
一个项目文件夹 + 一个 GitHub 仓库 + 一套 docs 文档 + 小步开发 + 阶段复盘 + 方法论沉淀
```

目标不是只做完一个产品，而是形成一套可复用的 AI 驱动设计与开发协作流程。
