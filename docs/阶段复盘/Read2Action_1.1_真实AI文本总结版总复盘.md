## 1. 版本定位

Read2Action 1.1 的核心不是做一个完整 AI 产品，而是在 1.0 mock 原型基础上，完成一次最小真实 AI 闭环。

1.1 验证的是：

用户输入文本  
→ 点击开始解析  
→ 进入 /parsing  
→ 服务端调用真实 AI  
→ 返回真实总结结果  
→ 进入 /result 展示  
→ 用户可保存结果  
→ 保存后可在最近 / 项目 / 详情页回看

本阶段重点是：

1. 真实 AI 接入是否能跑通；
2. AI 结果是否能进入现有产品结构；
3. 最小保存与回看闭环是否成立。

自动暂存、账号系统、数据库、多端同步、复杂归档逻辑，都不属于 1.1 范围。

## 4. 关键问题与尝试路径

1.1 过程中最重要的问题是 /parsing 页面曾重复请求 /api/analyze。

该问题表现为：

1. Network 中出现多条 analyze 请求；
2. 页面曾出现闪烁 / pending / 无法稳定进入结果页；
3. DeepSeek API 被重复调用；
4. 产生真实费用风险。

该问题被定义为 P0，因为它不是普通 UI bug，而是直接影响：

1. AI 调用费用；
2. 产品稳定性；
3. 用户体验；
4. 线上安全；
5. 用户信任。

排查过程中重点关注过：

1. /parsing 页面状态流转；
2. useEffect 是否重复执行；
3. React StrictMode 下的开发环境行为；
4. AbortController cleanup 是否导致请求中断后重新触发；
5. inFlightRef 是否只能防组件实例内并发；
6. commitRef 是否能防止重复提交；
7. queueMicrotask / 定时器 / stepsComplete 是否间接触发 runFetchOnce；
8. 组件重新挂载后是否再次消费 pendingAnalyzeText；
9. Turbopack FATAL 是否干扰本地判断；
10. dev script 是否需要切换为 next dev --webpack。

最终结论是：该问题不是单点原因，而是多个因素叠加导致，需要组合式防护和明确验收标准。

## 5. P0 Bug：重复请求与费用风险

1.1 接入真实 AI 后，/parsing 重复请求不再只是前端状态问题，而会直接转化为 API 调用成本、额度消耗和线上风险。

最终处理方式：

1. 保留 A+B+C 组合防重复请求方案；
2. 统一验收标准；
3. 将 dev script 改为 next dev --webpack，用于规避 Turbopack FATAL 对本地判断的干扰；
4. 用 Network 作为最终判断依据。

最终验收标准：

单次输入文本后，Network 过滤 analyze，只允许出现 1 条 POST /api/analyze，Status = 200，并成功进入 /result。

这个标准后续应作为所有付费 API 功能的基础验收方式之一。

## 6. API Key 安全策略

1.1 已建立 DeepSeek API Key 安全规则：

正确方案：

1. 前端不读取 API Key；
2. 前端只请求服务端 /api/analyze；
3. 服务端 Route Handler 读取 process.env.AI_API_KEY；
4. 服务端再调用 DeepSeek；
5. Vercel 使用环境变量存储 Key；
6. 本地使用 .env.local 存储 Key；
7. .env.local 不提交 GitHub。

长期规则：

1. 不要把 Key 发给 ChatGPT；
2. 不要把 Key 发给 Cursor；
3. 不要截图露出 Key；
4. 不要提交 .env.local；
5. 不要 git add -f .env.local；
6. 文档中只写变量名，不写真实 Key；
7. Vercel 中使用 Sensitive 环境变量；
8. 本地只放在 .env.local。

当前变量名为：

- AI_API_KEY
- AI_BASE_URL
- AI_MODEL

不是 DEEPSEEK_API_KEY。

## 7. 保存闭环与范围控制

1.1 完成了手动保存闭环，但也暴露了体验问题：

1. 用户必须在 /result 页面主动点击「保存到项目」；
2. 如果忘记保存，解析结果可能找不到；
3. 全局解析和项目内解析的归属关系还不清晰；
4. 最近列表到底展示正式笔记，还是同时展示未归档解析历史，需要后续重新定义。

当前决策：

1. 1.1 保留手动保存；
2. 1.1 不处理自动暂存 / 自动归档；
3. 自动暂存、自动归档、sourceContext、savedStatus、draft / temporary 状态等机制进入 1.2 讨论；
4. 当前只记录问题，不在 1.1 继续扩展范围。

这次范围控制是正确的：1.1 的目标是完成真实 AI 解析闭环 + 最小保存回看闭环，而不是一次性做完整知识库系统。

## 8. 设计师 × ChatGPT × Cursor 的 AI 协同工作流复盘

1.1 阶段不仅完成了真实 AI 文本总结能力的接入，也逐步沉淀出一套设计师与 AI 工具协同推进产品迭代的工作方式。

项目初期，设计师主要通过自然语言直接描述需求，但在实际执行中发现，单纯自然语言容易导致：

1. 目标不够明确；
2. 功能边界不清楚；
3. Cursor 容易自行发挥；
4. 修改范围可能扩大；
5. 验收标准不具体；
6. 出问题后不好追踪是哪一步偏了。

后续逐步调整为：

设计师提出问题和目标  
→ ChatGPT 分析产品目标、拆解功能边界、识别风险  
→ ChatGPT 将需求转译为结构化 Cursor 执行指令  
→ Cursor 完成代码实现  
→ Cursor 自检  
→ 设计师进行本地页面、Network、保存、刷新等人工验收  
→ ChatGPT 协助判断问题优先级和是否影响当前版本目标  
→ GitHub / Vercel 完成上线  
→ 线上再次验收  
→ docs 收尾  
→ 版本复盘与方法论沉淀

该流程的核心价值在于：AI 并不直接替代设计师决策，而是被纳入一个由设计师主导的协同系统中。

分工如下：

1. 设计师负责目标判断、体验判断、风险判断和最终验收；
2. ChatGPT 负责分析、拆解、转译、复盘和指令规范化；
3. Cursor 负责代码执行和基础自检；
4. docs 负责记录最终事实、版本结论和可复用方法。

这说明设计师不是简单「使用 AI 工具」，而是在设计一套 AI 协同产品迭代流程。

## 9. 防自嗨检查 / 现实校准机制

1.1 过程中加入了「防自嗨检查 / 现实校准」机制。

原因是：AI 工具显著降低执行成本后，项目过程中容易因为阶段性成果而过度兴奋，只关注「功能做出来了」，忽略真实产品中的费用、安全、稳定性、数据和用户体验问题。

后续每次进入新功能或新版本前，都需要先判断：

1. 这个问题是否影响当前版本目标？
2. 是否影响核心闭环？
3. 是否涉及费用 / 安全 / 数据风险？
4. 是否必须本版本解决？
5. 是否可以进入后续版本池？
6. 当前是否只是因为兴奋而想继续加功能？
7. 当前是否已经完成上线、验收、docs、复盘等收尾动作？

这套机制的价值是：避免因为 AI 执行速度变快，导致产品方向失控、功能不断膨胀、版本边界被打穿。

## 10. 当前沉淀的方法论

1.1 之后，当前形成了几个可复用方法：

### 10.1 版本推进流程

设定版本目标  
→ 明确功能边界  
→ 判断不做什么  
→ 生成 Cursor 执行指令  
→ Cursor 执行  
→ Cursor 自检  
→ 设计师人工验收  
→ ChatGPT 辅助判断  
→ 本地通过  
→ GitHub 提交  
→ Vercel 部署  
→ 线上再次验收  
→ docs 收尾  
→ 总复盘  
→ 沉淀方法论

### 10.2 Cursor 指令模板

后续给 Cursor 的指令建议固定包含：

1. 当前项目；
2. 当前阶段；
3. 本次目标；
4. 不做什么；
5. 必须保留什么；
6. 安全要求；
7. 验收标准；
8. 完成后自检；
9. 是否需要更新 docs；
10. 不允许擅自扩展范围。

### 10.3 API 功能验收清单

涉及付费 API、真实接口、云端能力时，必须检查：

1. Network 是否只有一次预期请求；
2. Status 是否为 200；
3. 是否成功进入目标页面；
4. 是否会重复扣费；
5. 是否有前端重复点击保护；
6. 是否有请求锁；
7. 是否有失败兜底；
8. API Key 是否只在服务端读取；
9. 线上环境变量是否正确；
10. Vercel 是否重新部署；
11. 线上是否再次验收。

### 10.4 版本复盘模板

每个版本结束后固定复盘：

1. 版本目标；
2. 实际完成内容；
3. 关键问题；
4. 做过哪些尝试；
5. 最终解决方案；
6. 遗留问题；
7. 工作流问题；
8. 后续怎么避免；
9. 能否沉淀为方法论；
10. 是否对作品集有价值。

## 11. 面向未来体验设计师工作方式的思考

Read2Action 不只是一个 AI 阅读总结工具，也可以作为一次「未来体验设计师如何与 AI 协同工作」的实践样本。

在 AI 能力越来越强的环境下，体验设计师的价值不再只是产出页面，而是更偏向于：

1. 定义问题；
2. 判断目标；
3. 拆解任务；
4. 明确边界；
5. 组织 AI 工具协作；
6. 将自然语言需求转化为结构化执行指令；
7. 设计验收标准；
8. 判断结果是否符合真实产品目标；
9. 控制安全、成本和体验风险；
10. 将过程沉淀为可复用方法论。

在这个项目中，设计师的角色不是被 AI 替代，而是成为 AI 工作流的组织者和判断者。

ChatGPT 负责分析和转译，Cursor 负责执行，设计师负责方向、判断、验收和沉淀。这个协作模式可以作为后续作品集里「AI 驱动设计工作流」的重要叙事方向。

## 12. 1.1 最终结论

Read2Action 1.1 已完成原定目标。

本版本将 Read2Action 从 1.0 假数据原型，升级为具备真实 AI 文本总结能力的最小可用版本。

具体成果包括：

1. 真实 AI 接入成功；
2. 服务端 API Key 安全方案成立；
3. 本地与线上均完成单次请求解析；
4. P0 重复请求风险已修复；
5. 真实 AI 结果可保存到本地；
6. 最近 / 项目 / 详情页回看闭环成立；
7. docs 已记录上线验收与安全策略；
8. 初步沉淀出设计师 × ChatGPT × Cursor 的 AI 协同产品迭代工作流。

现实校准：

1.1 是一个成功的最小 AI 闭环，但还不是成熟商业产品。

当前它能证明：

1. 可以从 0 到 1 搭建一个 AI 阅读总结工具；
2. 可以用 AI 工具辅助设计、开发、验收和复盘；
3. 可以发现并处理真实 API 的安全和费用风险；
4. 可以形成版本化推进和文档化复盘流程；
5. 可以沉淀面向未来体验设计师的 AI 协同工作方式。

但它还不能证明：

1. 这是完整商业产品；
2. 这是稳定可规模化系统；
3. 这是成熟知识管理工具；
4. 这是多用户 SaaS。

因此，1.1 更准确的定位是：

AI 驱动设计工作流实践项目的第一个真实 AI 功能版本。

## 13. 遗留问题

1.1 收尾后，仍然保留以下问题：

1. 解析结果必须手动保存；
2. 忘记保存可能导致结果找不到；

6. 缺少失败重试、错误提示、额度保护等更完整的 AI 产品机制；
7. 当前 AI 协同工作流仍然依赖人工截图和文字接力，后续可以继续优化文档化和模板化程度。

这些问题不影响 1.1 收尾，但应进入后续版本规划。
