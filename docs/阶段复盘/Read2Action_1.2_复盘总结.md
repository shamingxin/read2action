## 1. 版本目标回顾

1.2 的核心目标不是新增复杂能力，而是在 1.1「真实 AI 文本总结」基础上，解决一个真实工具必须具备的基础问题：

> 用户解析出的内容，不能只停留在一次性结果页里，而应该可以被暂存、找回、保存到项目，并形成基础记录生命周期。

因此，本版本围绕「不会丢内容」展开，重点补齐了以下体验：

1. 真实空状态，避免 mock 数据默认展示造成误导。
2. 全局自动暂存，未保存内容也能在最近找回。
3. /result 结果页可以保存到项目。
4. 未归档详情页也可以继续保存到项目。
5. 项目内解析可以自动归档到当前项目。
6. 原文对照展示真实输入原文，而不是占位内容。
7. 未归档详情页隐藏面包屑，避免点击 404。
8. /parsing 页面展示真实已用时间，不再固定或闪回。
9. 保持 /api/analyze 单次解析只有一次 POST。
10. 宽屏下结果页和详情页内容宽度收敛，小屏响应式正常。

最终结论：1.2 完成了「解析结果从生成到暂存、找回、归档」的基础闭环，Read2Action 从一次性 AI 生成工具，向可持续管理的内容工具迈进了一步。

## 2. 本阶段主要问题、探索过程与最终方案

### 问题一：mock 数据默认展示，导致真实产品状态不清晰

早期页面中存在 mock 内容默认展示，这在原型阶段可以帮助验证布局，但进入真实使用阶段后，会造成用户误解：

* 用户不知道哪些内容是真实生成的。
* 项目页、最近列表看起来像已有数据，但实际并非用户行为产生。
* 不利于验证真实空状态、真实保存链路。

探索过程：

最初我们考虑继续保留 mock 来维持页面完整度，但从实际产品体验出发，1.2 的目标是验证真实记录生命周期，因此 mock 默认展示会干扰验收。

最终方案：

* 默认进入真实空状态。
* mock 数据不再默认展示。
* 通过用户真实解析、真实暂存、真实保存来驱动页面内容出现。

沉淀结论：

> 原型阶段可以用 mock 撑页面，真实功能闭环阶段必须切换到真实数据状态，否则会影响验收判断。

---

### 问题二：解析结果未保存时容易丢失，缺少全局暂存机制

1.1 阶段用户完成解析后，如果没有主动保存，结果容易成为一次性内容。对一个工具产品来说，这会造成明显不安全感。

探索过程：

我们讨论了「保存」「归档」「最近」之间的关系，最终判断 1.2 不应该直接做云端数据库，而是在本地 localStorage 下先建立最小记录生命周期。

最终方案：

* 所有解析结果先进入全局自动暂存。
* 未保存内容可以在「最近」中找回。
* 用户后续可以从结果页或未归档详情页保存到项目。
* 项目内发起的解析结果可以自动归档到当前项目。

沉淀结论：

> 在没有数据库前，localStorage 仍然可以承担 MVP 阶段的记录生命周期验证；关键不是技术先进，而是先把用户感知上的「不会丢」跑通。

---

### 问题三：未归档详情页「保存到项目」无反应

这是 1.2-D 阶段的关键体验问题。用户从最近列表进入未归档详情页后，点击「保存到项目」没有反应。

问题本质：

* /result 页面保存链路已可用。
* 但详情页是另一条入口，状态来源和保存行为没有完全复用。
* 从体验上看，用户并不区分「结果页保存」和「详情页保存」，只会认为按钮失效。

探索过程：

我们先确认不是视觉问题，而是行为链路没有正确接上。随后要求 Cursor 聚焦修复该入口，不扩大到归档标签、状态展示等额外功能。

最终方案：

* 修复未归档详情页保存到项目逻辑。
* 保持 /result 保存与详情页保存行为一致。
* 验收时重点检查：从最近进入未归档详情页，再保存到项目是否成功。

沉淀结论：

> 只要用户能从多个入口看到同一类内容，就必须保证核心操作在每个入口都可用。不能只验主路径，也要验回流路径和二次操作路径。

---

### 问题四：原文对照展示不真实

在 1.2-E 阶段发现，结果页和详情页的「原文对照」存在占位或不真实展示问题。

问题本质：

原文对照是 Read2Action 的信任基础之一。用户需要知道 AI 总结对应的是哪段输入内容。如果展示的是假内容或占位内容，会削弱可信度。

探索过程：

我们没有把它当成纯展示优化，而是判断它属于结果可信度问题，应该在 1.2 内解决。

最终方案：

* /result 展示真实输入原文。
* 笔记详情页展示真实输入原文。
* 保证生成结果和原文来源可以对应。

沉淀结论：

> AI 工具不能只展示生成结果，还要展示输入依据。原文对照是信任机制，不是装饰模块。

---

### 问题五：未归档详情页面包屑点击 404

未归档笔记不属于具体项目，但页面上仍然出现项目面包屑，点击后会进入 404 或无效路径。

问题本质：

* 面包屑本质上表达信息归属。
* 未归档内容没有项目归属。
* 强行展示面包屑，会制造错误预期和错误入口。

探索过程：

起初可以考虑禁用面包屑或只隐藏可点击项，但从体验一致性看，未归档内容不应该出现项目路径。

最终方案：

* 未归档详情页隐藏整个面包屑区域。
* 已归档项目笔记继续保留项目面包屑。
* 保持信息架构表达清晰。

沉淀结论：

> 导航不是装饰，导航必须反映真实归属。没有归属关系时，不要为了页面完整而保留假导航。

---

### 问题六：解析页时间固定 20 秒，并出现闪回

/parsing 页面之前存在固定 20 秒或时间闪回问题，例如已经显示 17 秒，又突然回到 20 秒。

问题本质：

* 固定倒计时会让用户误以为 AI 正在按某个预估时间完成。
* 闪回会让用户感觉系统状态不稳定。
* 让大模型预测耗时没有必要，且会增加复杂度。

探索过程：

我们讨论过是否让 AI 预估解析耗时，但最终判断这是不必要的复杂化。MVP 阶段更应该展示真实已用时间，而不是伪预测。

最终方案：

* /parsing 页面展示真实已用时间。
* 不做 AI 预测耗时。
* 避免时间闪回。
* 只表达「系统正在处理」和「已经用了多久」。

沉淀结论：

> 状态反馈优先真实稳定，不要为了高级感引入不稳定预测。用户更需要确定性，而不是伪智能感。

---

### 问题七：宽屏下结果页和详情页内容拉伸过大

在最后总回归阶段发现，页面缩小时响应式正常，但宽屏拉伸时，内容区域过宽，右侧出现大量空白或阅读宽度失控。

问题本质：

* 工具类内容页需要控制阅读宽度。
* 宽屏不是无限拉伸，而是应该保持合理阅读容器。
* 这是影响专业感和可读性的体验问题。

最终方案：

* 约束结果页和笔记详情页内容最大宽度。
* 保持小屏响应式正常。
* 不做大规模 UI 重构，只解决当前视觉阻塞点。

沉淀结论：

> 响应式不只是小屏适配，也包括大屏阅读宽度控制。宽屏体验是工具产品专业感的重要组成部分。

## 3. 本阶段形成的固定工作流

### 工作流一：阶段开发必须拆小任务推进

本阶段多次验证，Cursor 在处理过大任务时容易卡顿、上下文混乱或中途停住。更稳定的方式是拆成小阶段：

1. 明确本轮只解决一个问题。
2. 给 Cursor 明确范围和禁止项。
3. 修复后立刻本地验收。
4. 通过后再进入下一项。
5. 每完成一个稳定节点再提交 commit。
6. 长对话或上下文变重时及时新开 agent 接力。

适用场景：

* 1.3 登录
* 1.3 数据库
* 1.3 数据迁移
* 1.4 UI 重构
* 后续文件上传、链接解析等复杂功能

固定模板：

```text
请只处理当前问题，不新增功能、不重构无关代码。
当前目标是：XXX。
验收标准是：
1. XXX
2. XXX
3. XXX

禁止修改：
1. XXX
2. XXX
3. XXX

完成后请返回：
1. 修改了哪些文件
2. 为什么这样改
3. 如何验收
4. 是否影响已有功能
```

---

### 工作流二：每个功能必须同时验主路径和回流路径

1.2 的「保存到项目」问题说明，只验 /result 主路径不够，还要验用户从「最近」回来的路径。

后续验收应固定包含：

1. 首次进入路径。
2. 结果页主路径。
3. 刷新后路径。
4. 最近列表回流路径。
5. 项目详情路径。
6. 异常状态路径。
7. 宽屏和窄屏展示。

沉淀为一句话：

> 只要一个功能有多个入口，就必须逐入口验收，不能只验最顺的那条路径。

---

### 工作流三：每次上线前固定走质量闸门

1.2 收尾阶段形成了较稳定的上线流程：

1. `npm run test`
2. `npm run lint`
3. `npm run build`
4. `git status`
5. commit 检查
6. push GitHub
7. Vercel 自动部署
8. 线上验收
9. 版本冻结

这套流程后续应固定化，作为每个版本的上线前 checklist。

---

### 工作流四：体验问题先判断是不是 P0/P1，再决定是否纳入当前版本

1.2 中出现过一些可以优化但不一定要当场做的点，例如：

* 最近列表状态标签
* 更完整的归档说明
* UI 大重构
* SEO 官网化

最终采取的策略是：

* 阻塞主链路的，纳入当前版本。
* 影响信任或保存闭环的，纳入当前版本。
* 只是增强表达或视觉优化的，进入后续版本。
* 不为了顺手优化破坏版本收口。

沉淀结论：

> 版本后期最重要的是收口，不是把所有想法都做完。能冻结的版本，才是真正可持续迭代的版本。

## 4. 从体验设计师 / 验收者 / 方案主持者角度的复盘

### 做得好的部分

1. 能及时发现真实使用中的体验问题
   例如未归档详情页保存无反应、面包屑点击 404、时间闪回、宽屏空白等，都是在真实操作中发现的，而不是只看代码结果。

2. 能坚持从用户感知判断问题优先级
   比如原文对照不真实，本质上不是小 UI 问题，而是信任问题；未保存内容能否找回，本质上不是存储问题，而是安全感问题。

3. 能在版本后期主动收口
   最近列表状态标签、UI 大重构、SEO 官网化都没有强行塞进 1.2，这一点是正确的。

4. 能把自己定位为体验负责人，而不是被 Cursor 牵着走
   Cursor 负责实现，但版本范围、验收标准、问题优先级由人来判断。

---

### 存在的疏漏

1. 前期对「记录生命周期」的入口枚举不够完整
   一开始更关注 /result 保存到项目，但没有第一时间把「最近进入未归档详情后再保存」作为同等重要路径，导致后期发现按钮无反应。

2. 对导航归属关系的检查稍晚
   未归档页面不应该展示项目面包屑，这个问题是在体验验收中发现的。后续信息架构类页面需要提前检查「它属于哪里」「用户点击会去哪里」。

3. 对展示内容真实性的验收不够早
   原文对照一开始没有作为强校验项，后续才发现不是真实原文。AI 产品中，输入、输出、来源之间的对应关系应提前纳入验收清单。

4. 宽屏体验没有在早期验收清单中固定下来
   前面主要关注功能和小屏/正常屏，最后才发现宽屏内容宽度问题。后续所有详情类页面都应该固定检查宽屏阅读宽度。

5. 对 Cursor 的任务粒度仍有几次偏大
   部分任务一开始描述较长，容易导致 Cursor 卡顿或执行过多。后续应更坚决地分段推进，每次只解决一个明确问题。

---

### 对体验设计师角色的校准

这阶段说明，体验设计师在 Vibe Coding 中不是只负责「看页面好不好看」，而是要承担三个角色：

1. 产品主持者
   判断当前版本应该做什么、不做什么，以及什么时候收口。

2. 验收负责人
   不只是看结果是否出现，而是验证每条路径是否符合真实用户行为。

3. 体验架构师
   判断信息归属、状态反馈、数据生命周期、信任机制是否成立。

对后续作品集表达也有价值：

> 本项目中，设计师不是把需求交给 AI 和 Cursor 执行，而是通过版本边界、路径验收、问题分级和复盘机制，持续主持一个 AI 辅助开发流程。

## 5. ChatGPT / Cursor / 人的协作问题与解决方式

### 遇到的问题

1. 无法直接读取本地代码和线上环境
   很多判断依赖用户截图、Cursor 返回、终端结果和线上验收描述。

2. 长周期项目上下文复杂
   1.2 涉及多个小阶段、多次 Cursor 接力、多次问题修复，如果不及时收口，容易混淆已完成和待处理事项。

3. 需要在「继续优化」和「版本冻结」之间不断做判断
   很多问题都看起来可以顺手优化，但不一定适合纳入当前版本。

4. 需要把研发问题转换成设计师可理解、可决策的语言
   例如 commit、push、Vercel、localStorage、POST 次数、build 等，都需要翻译成对应的产品状态和风险。

---

### 解决方式

1. 通过截图和命令结果做外部证据确认
   每次判断 push、部署、验收，都基于可见结果，而不是直接假设完成。

2. 使用阶段接力文案保持上下文连续
   每次新开 ChatGPT / Cursor 对话前，都提炼当前版本、完成项、禁止项和下一步，减少上下文丢失。

3. 强制执行「不扩大范围」原则
   在版本后期，只处理 P0/P1 或明显影响闭环的问题，避免 1.2 无限延期。

4. 将技术状态翻译为产品状态
   例如：

   * git clean = 本地没有未提交改动
   * push 成功 = GitHub 已同步
   * Vercel Ready = 线上版本已部署
   * Network 只有一次 POST = 没有重复请求费用风险

沉淀结论：

> AI 助手在项目中不应只是回答问题，而应帮助用户维持项目节奏、风险判断和版本边界。

## 6. 可沉淀为后续 1.3 / 1.4 的方法论

### 方法论一：先定义数据生命周期，再做功能页面

1.2 的核心不是页面，而是生命周期：

输入文本 → 解析中 → 生成结果 → 自动暂存 → 最近找回 → 保存到项目 → 项目内管理

后续 1.3 做登录和数据库时，也必须先定义生命周期：

未登录数据 → 登录后数据 → 本地数据迁移 → 云端保存 → 跨设备同步 → 权限校验 → 退出登录后的状态

结论：

> 复杂功能不要先画页面，先画数据和状态怎么流动。

---

### 方法论二：AI 产品必须验「输入-处理-输出-保存」全链路

后续每个 AI 能力都应检查：

1. 输入是否真实。
2. 请求是否只发一次。
3. 处理中状态是否稳定。
4. 输出是否可信。
5. 原文/来源是否可追溯。
6. 结果是否可保存。
7. 保存后是否可找回。
8. 找回后是否可继续操作。

这套方法可用于：

* 1.3 云端保存
* 1.4 视觉重构后的回归
* 后续链接解析
* 后续文件上传
* 后续多模型能力

---

### 方法论三：版本越到后期，越要反新增功能

1.2 后期有多个可以继续做的点，但最终没有扩展，这保证了版本顺利上线。

后续固定原则：

1. 版本前期可以探索。
2. 版本中期可以修正。
3. 版本后期只收口。
4. 上线前只处理阻塞。
5. 线上通过后立刻冻结。

结论：

> 版本管理的能力，不是把所有问题都做完，而是知道哪些问题必须现在做，哪些问题应该延后。

---

### 方法论四：Cursor 协作要使用「小任务 + 禁止项 + 验收标准」

后续给 Cursor 的任务不应是「帮我把 1.3 做完」，而应该拆成：

1. 只建认证方案文档。
2. 只设计数据库 schema。
3. 只接 Supabase 初始化。
4. 只做登录页。
5. 只做登录状态识别。
6. 只做本地数据迁移。
7. 只做云端保存。
8. 只做线上部署验收。

每一步都要明确：

* 目标
* 范围
* 禁止项
* 验收标准
* 回滚风险

结论：

> AI 编程不是一次性托管，而是设计师主持下的分段执行。

---

### 方法论五：体验验收要从「页面检查」升级为「路径检查」

页面检查只看这个页面是否正常。
路径检查要看用户从哪里来、做什么、去哪里、能不能回来。

1.2 中最典型的问题是未归档详情页保存按钮无反应。单看 /result 页面没问题，但路径一变，问题就暴露了。

后续验收固定包含：

1. 新用户路径。
2. 老用户路径。
3. 刷新路径。
4. 返回路径。
5. 异常路径。
6. 空状态路径。
7. 多设备或多状态路径。
8. 宽屏 / 窄屏路径。

结论：

> 真实体验不是页面集合，而是路径集合。

## 7. 对 1.3 / 1.4 的提前建议

### 1.3 登录与数据库阶段建议

1.3 不建议一上来直接写代码。应先完成：

1. 账号体系范围定义
   例如是否只做邮箱登录、GitHub 登录、Google 登录，是否允许未登录继续使用。

2. 数据库表结构设计
   至少包括 users、projects、notes / records 等核心表。

3. 本地数据迁移策略
   1.2 已经有 localStorage 数据，1.3 必须考虑用户登录后是否迁移本地数据。

4. 权限边界
   用户只能访问自己的项目和笔记，不能通过 URL 访问别人的数据。

5. 未登录和已登录状态差异
   例如未登录时是否仍可解析、是否提示本地保存、登录后如何同步。

6. 最小上线范围
   1.3 只解决账号和云端保存，不应该同时做大 UI 重构、多模型、SEO 或文件上传。

建议 1.3 版本目标：

> 从本地保存升级为账号体系和云端保存，让用户在登录后可以跨设备找回项目和解析记录。

---

### 1.4 体验与视觉优化阶段建议

1.4 更适合处理：

1. 视觉系统统一。
2. 首页信息表达优化。
3. 结果页阅读体验优化。
4. 项目页和详情页结构优化。
5. 最近列表状态标签重新评估。
6. 官网化 / SEO 基础配置。
7. README 更新为正式项目说明。
8. 品牌感和作品集展示包装。

注意：1.4 不应和 1.3 混在一起。
账号数据库是基础能力，视觉重构是体验表达，二者同时做会增加回归风险。

---

## 8. 本版本最终结论

1.2 是 Read2Action 从「能生成」走向「能管理」的重要版本。

它解决的不是单个按钮或页面问题，而是建立了最基础的记录生命周期：

> 解析结果可以生成、暂存、找回、保存、归档，并在项目中继续查看。

同时，本阶段也沉淀出更成熟的 AI 协作开发方式：

1. ChatGPT 负责规划、拆解、验收标准和复盘。
2. Cursor 负责具体代码实现。
3. 设计师负责体验判断、范围控制和最终验收。
4. 每个版本通过 docs、commit、部署和线上验收完成闭环。

1.2 结束后，Read2Action 已具备进入 1.3 的基础条件。

下一阶段不应急着开发，而应先完成 1.3 PRD 和数据库方案设计，再拆小任务交给 Cursor 执行。
