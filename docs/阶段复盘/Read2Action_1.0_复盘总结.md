## 1. 1.0 阶段定位

- **产品**：Read2Action — AI 知识整理与行动转化工具 **1.0**。  
- **工程形态**：**mock 数据 + 本地/前端状态** 的 **可部署 Web 原型**，六页主流程完整、视觉接近真实 SaaS。  
- **价值主张**：在 **不接真实 AI、不接数据库、不做登录** 的边界下，完成 **从构想到线上可点** 的闭环，为 **1.1「真实 AI 文本总结版」** 打地基。  
- **诚实边界**：1.0 **不是**「已具备生产级 AI / 数据」的产品；**mock 为预期，非 bug**。

## 3. 关键决策（摘要）

- **1.0 冻结**：不扩展为「顺手做真实 AI / 数据库 / 登录」。  
- **验收以事实为准**：**9.3** 以 **文档 + 浏览器表现** 为准，而非单一旧版交互稿。  
- **上线与诚实叙事**：先 **mock 可演示、可部署**，再规划 **1.1**。  
- **1.1 收窄**：**只做** 用户文本 → AI 结构化总结 → 结果页真实展示；**不做** 链接全文抓取、登录、数据库、多用户、文件上传、插件、复杂知识库、方法论自动生成等（详见核心文档）。

---

## 4. 暴露问题（复盘承认）

1. **文档分散**：早期多文件并行，**接力成本高** → 已通过 **`01`～`07`** 收敛。  
2. **长对话上下文成本**：Cursor 单线程过长易 **丢假设、重复解释** → 依赖 **新开对话 + @ docs**。  
3. **Cursor 易自作主张**：边界未写死时倾向 **加依赖、扩 scope** → 需 **「不做清单」+ 小任务 + 验收**。  
4. **人工验收不可替代**：**构建通过 ≠ 产品对**；主流程与观感仍以 **人 + 走查** 定义完成。  
5. **1.0 仍是 mock 演示版**：作品集叙事须 **边界诚实**，避免夸大成「完整 AI 产品」。

## 5. 实际 AI 设计开发工作流（1.0 真实路径）

```text
产品构想
  → PRD / MVP 与功能确认
  → 低保真结构与信息架构
  → 高保真与 Figma 6 页终稿（设计说明可对照）
  → Cursor 分阶段实现（mock + 本地状态 + 组件化）
  → 用户人工验收（主流程 + 线上）
  → GitHub / Vercel 部署
  → 复盘沉淀写回 docs（含 `07` §11 与本归档）
```

## 6. ChatGPT / Cursor / 用户 — 三方分工

| 角色 | 主要职责 | 本项目中的体会 |
|------|----------|----------------|
| **ChatGPT** | 结构化需求、PRD 与流程、复盘框架、风险清单 | **强**于「写清楚」；结论需 **回写 docs** 与仓库真值对齐。 |
| **Cursor** | 依 docs + Figma **落地代码与 UI** | **强**于吞吐；需防 **扩 scope**；适合 **小步、可验证** 任务。 |
| **用户** | **拍板范围**、**验收**、**合并/部署**、**写进 docs 的共识** | **最终定义权**；对 mock 与 bug 的区分 **不可替代**。 |

---

## 7. AI 是否提效 — 阶段性判断

- **提效明显**：规格文档化、任务拆分、样板 UI / 组件由 Cursor 承担时，**整体快于纯手写**。  
- **提效被侵蚀**：长会话、需求只在聊天改口 **未同步 docs** 时，**返工与纠偏**会吃掉收益。  
- **结论**：提效 **高度依赖「docs 真值 + 小步验收」**；不是「全自动少动脑」。

## 8. 可复用方法论（简表）

| 方法 | 要点 |
|------|------|
| **先闭环再高级** | mock 主链路 + 可部署，再谈真实 AI / 账号 / 落库。 |
| **docs 管理上下文** | 共识进 `01`～`07`；聊天为推导，**文档为真值**。 |
| **小任务驱动 Cursor** | 可验证、可回滚；拒绝一次「做完整个项目」。 |
| **重要节点 Git 提交** | 阶段与文档变更 **单独 commit**，责任边界清晰。 |
| **阶段性防自嗨** | 对照 PRD：**是否超范围？mock 是否被误报为 bug？** |

## 9. 对 1.1 的启发

- **先方案后代码**：在 **不改 1.0 冻结** 前提下，优先 **AI 接入方案、数据结构、Prompt、API 契约** 文档化，**再立项开发**。  
- **继续用边界换速度**：1.1 已冻结 **「文本总结版」** — **不做抓取与账号体系**，避免 1.1 拖成小 2.0。  
- **把「链接输入」当产品沟通问题**：用 **轻提示** 管理预期，而非在 1.1 引入抓取复杂度。  
- **验收前置写清**：1.1 Done 的 E2E 应写进 **`03`** 立项附件，避免上线后争议「算不算 AI」。
